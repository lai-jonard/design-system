la-tabs

Single-selection tab list. One tab is active at a time; its associated panel is visible, the others are hidden. Four variants cover the navigational hierarchy — segmented (default, modern primary tabs), underline (older line-tab convention, kept for legacy chrome), pill (section-level sub-nav), and vertical (rail-as-layout for settings / deep nav). Three components compose the primitive — la-tabs owns active state, la-tab is a trigger, la-tab-panel is content. Full WAI-ARIA tablist keyboard model is built in (Arrow keys axis-aware, Home / End).

Hierarchy

Pick the variant that matches the navigational tier — most prominent at the top, smallest utility role at the bottom. Vertical sits orthogonally; it's a layout choice rather than a step on the prominence ladder.

TierVariantRoleTypical surface
1 segmented Default tab pattern. Modern primary navigation between top-level views, plus view-mode toggles where a clear "switching" affordance helps. Top-level views — "Overview / Documents / Audit", entity detail pages, view-mode toggles like "Week / Month / Year"
2 underline Older line-tab convention. Still works — reach for it when matching legacy doc/admin chrome, or when a clear bottom-border separator helps page hierarchy read. Doc-style page tabs, classic admin chrome
3 pill Section-level. Sits inside a card or panel, navigating between subsections of one larger area. Card sub-nav, dashboard widget tabs, settings sub-sections
— vertical In-page tab pattern, vertical layout. Panels swap in place; no URL routing. Carries role="tablist" semantics. Not for app-shell navigation — that's la-sidebar's job. Settings dialogs, tabbed inspectors inside cards/modals, deep nav inside a single page area

Rules from the hierarchy:

  • Default to segmented. Reach for it first for any primary tab pattern. Drop to underline only when matching legacy chrome or when the bottom-border separator is doing real layout work.
  • Don't nest the same variant in itself. Segmented-inside-segmented reads as broken hierarchy. If you need second-level tabs inside a segmented tab's panel, drop down to pill to signal the level change.
  • Don't pair siblings of different tiers in the same band. Pick one tier per visual area.
  • Vertical isn't a step on the prominence ladder — it's a layout choice. Use it when the navigation needs to stay visible alongside content (settings, deep nav), not when you want a "smaller" tab style.

la-tabs[variant=vertical] vs la-sidebar — when to use which

Both are vertical lists with single selection, but they serve different scopes:

  • la-tabs[vertical] — in-page tab pattern. Panels swap in place; no URL routing. Lives inside a card, modal, settings dialog, or content area. Carries role="tablist" + tab/panel ARIA semantics. Variants surface in-page concerns (size, density).
  • la-sidebar — app-shell navigation. Items route to different views/URLs. Lives at the edge of the app. Has app-shell-specific variants (collapsed icon-only, brand area, footer slot, multi-section grouping).

Industry note: the same split exists across Material UI, Radix, Ant Design, Chakra, and most production SaaS apps (Linear, Stripe, Notion, Figma). App sidebars are always their own component; the tabs component supports vertical orientation for in-page use. The variant set on each is intentionally different — sidebar's collapsed-icon mode would be meaningless inside a settings panel, tab sizes would be meaningless on a full-height app rail.

Properties

PropertyAttributeTypeDefaultDescription
active active string '' Currently active tab value — matches a `la-tab[value]` child. Reflected.
variant variant TabsVariant 'segmented' Visual treatment + layout. Default `segmented` is the modern primary tab pattern — bordered group with a tinted rail and a lifted active segment. `underline` is the older line-tab convention, still supported for legacy page chrome. `pill` is for section-level sub-nav inside a card or panel. `vertical` is the layout variant — rail sits as a column to the left, panels render to the right (settings, deep nav).
block block boolean false Full-width mode. When true, the rail fills its parent's width and child tabs distribute equally — useful in modals, constrained widths, and full-bleed app chrome where a fixed-size segmented control with equal segments is preferred. No-op for the vertical variant.
size size TabsSize 'default' Tab size — matches la-button's metric grid so tabs and buttons sit on the same vertical alignment. `default` is 32px, `sm` is 28px, `lg` is 44px. Currently surfaces variant-specific sizing on the pill variant; other variants stay at default.

Usage examples

Basic — two tabs

<la-tabs active="formation"> <la-tab slot="triggers" value="formation">Formation</la-tab> <la-tab slot="triggers" value="board">Board</la-tab> <la-tab-panel value="formation">…</la-tab-panel> <la-tab-panel value="board">…</la-tab-panel> </la-tabs>

With icons — leading Phosphor glyph per tab

<la-tab slot="triggers" value="overview"> <i slot="icon" class="ph ph-house"></i> Overview </la-tab>

Listening for active-change

const tabs = document.querySelector('la-tabs'); tabs.addEventListener('la-change', (e) => { // e.detail.value is the newly-active tab's value console.log(e.detail.value); });

Programmatic active — set from outside

// Attribute tabs.setAttribute('active', 'board'); // Property tabs.active = 'board';

Keyboard navigation — built in, no wiring

// Tab focuses the active trigger in the rail // ← / → cycles to previous / next, wraps at ends // Home / End jumps to first / last // Enter / Space activates the focused tab

In context — wrapping la-list-item rows

<la-tabs active="contracts"> <la-tab slot="triggers" value="contracts">Contracts</la-tab> <la-tab slot="triggers" value="consulting">Consulting</la-tab> <la-tab-panel value="contracts"> <la-list-item>Alex Chen</la-list-item> <la-list-item>Jordan Reyes</la-list-item> </la-tab-panel> <la-tab-panel value="consulting"> <la-list-item>Morgan Lee</la-list-item> </la-tab-panel> </la-tabs>

Notes

Three-component composition — la-tabs owns state, la-tab triggers, la-tab-panel holds content. Each has a distinct ARIA role. Alternative single-component API was rejected to keep the roles explicit.
WAI-ARIA keyboard model is built in — Arrow Left / Arrow Right cycle triggers with wrap, Home / End jump to first / last, Enter / Space activate. Roving tabindex (active tab = 0, others = -1) so Tab enters the rail once, not once per tab.
Click-on-active is a no-op — re-clicking the already-active tab does not re-dispatch la-change. Matches the WAI-ARIA tab convention and prevents event-listener spam.
Active colour is --la-color-primary — aligns with the design system rule that the primary colour marks interactive / active state. The founder-flow prototype pages use a one-off cobalt blue; that's drift — the migration sweep will flip them to match.
2px inset underline, not border-bottom — the active indicator is a box-shadow: inset 0 -2px 0. Avoids the 2px layout shift between active and inactive triggers that a border-bottom would cause.
Long-label overflow — v1 keeps labels on one line and lets the rail container overflow at narrow widths. Auto-scroll controls and a more-menu are out of scope. Revisit if a real content-volume problem surfaces.
Inset focus ring — the focus ring uses a negative outline-offset to sit inside the trigger. Flush-to-siblings tabs would overlap with an outset ring. Matches the inset-ring decision made for la-list-item.
Panels are focusable — la-tab-panel carries tabindex="0". WAI-ARIA recommends this so keyboard users can Tab into a panel when it contains no other focusable content. If your panel has its own focusable children, you can drop the host tabindex on that instance.
No panel enter / exit motion — the active panel becomes visible instantly, others get the native hidden attribute. Fade-in via motion.css's .anim-fade-in utility can be added per-consumer if needed.
