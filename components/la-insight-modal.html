la-insight-modal

The detail and action surface for a single insight. la-insight-card surfaces an issue at a glance; this modal takes over when the user wants the full story — explanation, all dates, entity context, source document, related insights, and the resolution action. Rendered as a fixed-position backdrop with a scrollable panel. Dismissable via close button, backdrop click, or the Dismiss footer button.

Two resolution mechanics driven by action-type: confirm (synchronous "Mark as complete" — for obligations the system cannot verify) and upload (inline la-upload zone — for document issue and missing document insights).

A grouped mode activates when the subItems prop is set — for the same obligation applying to multiple entities (e.g. three unsigned employment contracts). An entity tab strip renders below the header; each tab carries its own resolution state. The group resolves when all tabs are resolved.

Grouped mode

Activated when the subItems JS property is set. Applies when the same obligation covers multiple entities — for example, unsigned employment contracts across three employees.

Structure in grouped mode
  • Entity tab strip — rendered below the header, one tab per sub-item. Active tab highlighted; resolved tab shows a check icon.
  • Shared body sections — "Why this matters", source document, and related insights render once (obligation-level context, not per-entity).
  • Tab panel — per-entity dates and resolution widget (confirm button or upload zone) replace the single-insight body sections.
  • Footer — Dismiss only. Primary resolution action moves into each tab panel.
  • Group resolved — when all tabs are resolved, the modal transitions to the standard success footer.
Resolution flow
  1. User opens modal — tab strip shows all entities.
  2. User works through each tab — resolves each sub-item independently.
  3. Resolved tabs show a check icon in the tab label.
  4. When the last sub-item is resolved: la-insight-resolved fires and the modal transitions to the success footer.

Each individual resolution fires la-subitem-resolved with { subItemId }. The consumer (LiveView) updates the backend after each event. Tab resolved state is tracked internally — no external state management needed.

Action types

confirm — synchronous resolution

Used for action_required insights — obligations the system cannot verify (e.g., filing an annual statement, registering a trademark). The "Mark as complete" footer CTA resolves the insight immediately. A "Reopen" CTA appears on the resolved modal so the user can correct accidental resolutions.

upload — document resolution with processing

Used for document_issue and missing_document insights. Renders an inline la-upload zone in the modal body. The modal shows a processing state after server receipt; the user can dismiss and continue working. The global processing widget tracks the outcome. On success, the insight auto-resolves and cascade resolution re-evaluates all insights referencing the same document. On failure, the modal re-opens with a verification error and re-upload option. No Reopen CTA on the resolved state — the system-verified state is always correct.

Events
Event Detail When
la-insight-close — Close button, Dismiss footer button, or backdrop click. Also fires on Escape (native dialog behaviour).
la-insight-resolved { headline } User resolves a single insight (confirm or upload). In grouped mode, fires when the last sub-item is resolved. The component sets resolved = true and transitions to the success footer.
la-insight-reopen { headline } User clicks "Reopen" on a resolved confirm-type modal. Consumer drives the backend call; component sets resolved = false.
la-insight-upload { source: 'local'|'drive', headline, subItemId?: string } User selects or drops a file (local) or clicks the Google Drive button. In grouped mode, subItemId identifies which entity's tab triggered the upload. Consumer drives the upload pipeline; set upload-state="processing" on receipt, 'error' on failure, resolved on success.
la-subitem-resolved { subItemId: string } Grouped mode only. Fires each time an individual entity's tab is resolved — before the group-level la-insight-resolved. Consumer calls the backend to mark that sub-item resolved. Tab resolved state is tracked internally — no need to pass it back in.
la-insight-navigate { id } User clicks a related insight link. Consumer navigates to /insights/{id} — closes this modal, opens the target insight's modal on its owning legal area page.

Notes

Source document CTA — disabled until Sprint 6

The "View document" CTA renders as a disabled ghost button until Sprint 6 (Document System) builds /documents/{id}. Pass source-document now — the name is shown and the id is stored. Sprint 6 will wire the navigation without an API change.

Consumer drives upload state

The component fires la-insight-upload when the user selects a file or clicks Google Drive. The consumer (LiveView) owns the upload pipeline — set upload-state="processing" on receipt, 'error' on verification failure (pass verification-error with the backend's message), and resolved on success. The component does not manage async state internally beyond the confirm-completion animation.

Cascade resolution is a backend concern

When the user uploads through this modal, the backend re-evaluates all insights referencing the same document, regardless of legal area. Related insights that pass verification auto-resolve via LiveView server-push. The component's role is to show the cascade note ("Uploading a corrected document will re-evaluate all issues") and fire la-insight-upload — the rest is backend propagation.

Reopen — confirm insights only

Confirm-completion insights (user-asserted) show "Reopen" in the resolved footer so accidental resolutions can be corrected. Document upload insights (system-verified) do not — the system's verified state is always authoritative. If the user uploaded the wrong document, they re-upload; the system recalculates.

Grouped mode — internal tab state

The resolved state of each entity tab is tracked internally — the consumer does not need to pass it back in. Each resolution fires la-subitem-resolved immediately; the consumer calls the backend. If the page re-renders (e.g. LiveView push), the consumer can pre-populate initial resolved state by setting resolved: true on any GroupSubItem — the component respects the initial value on first render only.

Grouped mode — upload-state is shared

The upload-state prop is a single shared value in grouped mode — it reflects the state of whichever upload is currently in flight. The consumer should track which sub-item is being processed (via the subItemId in la-insight-upload) and update upload-state accordingly.
