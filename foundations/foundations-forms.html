Foundations / Forms

Forms

Composition rules for the form surfaces sprints 2 and 4 build — the dashboard welcome form, the unlock flow's text and multi-field steps, and any future form-bearing surface. The system has primitives for the field types (la-input, la-select, la-checkbox, la-switch); this page covers what sits around them — labels, hints, error text, submit rows, validation rhythm.

Field anatomy

One form field stacks four things vertically: label · control · hint · error. Hint and error are mutually exclusive on the same field — when error replaces hint, the slot's height stays roughly the same so the form doesn't jump.

Use the legal name on your incorporation documents.
LabelAbove the control. --la-font-size-base / --la-font-weight-semibold / --la-color-text. Sentence case. Verb-free — describes the field, not the action.
Required markerSingle * in --la-color-urgent appended inline to the label, never on the input. Mark every required field — don't mark optional ones (some forms invert this; we do not).
ControlReach for an la-* primitive first (la-input, la-select, la-checkbox, la-switch). Native controls only when the system primitive doesn't fit — see Native form controls.
HintOne short sentence under the control. --la-font-size-md / --la-color-text-muted. Only when the field needs context the label can't carry alone — never restate the label.
ErrorReplaces hint on validation failure. --la-font-size-md / --la-color-urgent-text. Lead with a Phosphor warning glyph (ph-fill ph-warning-circle) so colour isn't the only signal.
Why label-above-control, never label-beside? Stacked labels survive long copy at any width and translate cleanly to tablet/mobile. Side-by-side labels collapse awkwardly when the control grows or shrinks. The Sprint 2 welcome form and Sprint 4 unlock form both follow this rule.

Field width

Every form lives inside a constrained surface. We don't stretch text inputs to the full container width when a sensible cap exists — long text inputs read as expensive infrastructure.

Field typeWidthReason
Short text (name, email, code)240–320pxComfortably fits the value with margin. Never full-width on a wide surface.
Address line100% of column (capped at 480px)Real-world addresses run long.
Long text / description100% of column (capped at 560px)Match optimal reading line length.
Single-select / dropdown240–360pxCaps trigger width — option list still expands as needed.
Number (count, year, percent)80–120pxNumeric values are short; wider feels off.
Date140–180pxFormat-aware width.

Caps are caps — if the column is narrower, the field shrinks with it. Set max-width on the field, not width, so the form stays responsive.

Multi-field grouping

When two or three fields belong together (street + city + zip; first name + last name), pair them in a single row at desktop widths. Stack them when the column gets too narrow — don't try to fit four fields side-by-side regardless of viewport.

Use real <fieldset> + <legend>. Group related fields semantically, then style with border: none. Screen readers announce the legend before each grouped field — that's the accessibility win that's hard to replicate with a styled heading.

Validation states

Validate on blur for individual fields. Validate on submit for the whole form. Do not validate on every keystroke — it punishes the user mid-type. Errors live with the field they describe, not as a banner on top.

Five digits. We use it to confirm jurisdiction.
TriggerWhat runsWhere the result shows
Blur on a single fieldThat field's validatorsInline below the field — replaces hint with error text.
Submit on the formAll validatorsErrors appear under each failing field. Submit re-enables once everything passes.
Backend validationServer responseEither a step-level message (la-unlock-panel failed state) or a la-toast variant="danger" if the entire submit failed.
Submit disabled until valid. The submit button stays disabled until every required field passes its validator. Don't let users click submit only to be told their form is incomplete — disable it and let the absence speak.

Submit row

Right-aligned. Primary action at the right edge, secondary action(s) to its left. One thin divider above so the row reads as a footer.

Cancel Save
Verb-first labels"Save", "Send", "Submit". Sentence case. Match the verb to the outcome the user is causing.
One primary, optional secondaryPrimary lives at the right. Cancel/back buttons sit to its left as variant="ghost" — never a second primary.
Modal forms keep the same shapeThe submit row sits at the bottom of the modal body, separated by the same border-top divider.

Native vs la-* primitive

Reach for the la-* primitive first. Use a native control when the primitive doesn't fit and a separate component would be over-engineered.

NeedDefaultFall back to native when
Text inputla-inputNever — always use the primitive.
Email / URL / numberla-input type="email|url|number"Never — always use the primitive.
Single-select dropdownla-select + la-optionInside an embedded LiveView form whose serialiser only handles native <select>; track to migrate.
Booleanla-checkbox or la-switchInside a la-data-table selection cell — the table primitive owns the box.
Long-form textNative <textarea> styled by la-base-root.cssSystem primitive doesn't exist yet; use native + base styling.
Yes/no choicePending la-radio-group / la-card-pickerToday: native <input type="radio"> with shared name. Schedule the primitive build.
File uploadla-upload-boxNever — always use the primitive.
