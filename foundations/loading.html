Foundations · Loading

Loading

The system pattern for "the user has waited and we owe them feedback." Covers every place we communicate "something is happening" — from a 200ms button press to a 30-second AI document analysis. This page is the decision layer above the components — pick a category here first, then build, not the other way round.

Principles

Five rules, each with its because. They govern every loading decision in the product.

Match the indicator to the wait, not the action A 60ms click and a 30s analysis are different problems. The indicator reflects how long the wait is and whether we know its length — not what the user clicked.
The first 1000ms is psychological, not visual Never show a loading state for under 300ms — it flickers. Once shown, hold ~400ms minimum. The design system records this rule; engineering owns the runtime.
Skeleton for content, spinner for action Skeletons let the user see the page take shape — right for content fetches. Spinners answer "did my click land?" — right for buttons. Don't mix the two.
Show progress whenever you can show progress If the system knows how far through it is — bytes, count, percent — use a determinate indicator. Indeterminate spinners are a fallback, not a default.
Loading owes the user a recovery path Every loading state must have a designed timeout / failure variant. A spinner with no failure path is a trap.

Thresholds

Industry research (Nielsen, Apple HIG) is consistent. The right indicator depends on the wait, not the action.

WaitPerceived asIndicator
< 100 msInstantNone
100–1000 msResponsive but waitingNone — premature spinners feel slower
1–10 sDefinite waitSpinner / skeleton / inline progress
> 10 sLong taskDeterminate progress + time estimate
> 30 sBackground taskDefer to a widget the user can leave (la-processing-widget)
The rule that matters. Never show a loading state for under 300ms — it flickers and makes fast operations feel broken. Either no indicator at all, or a minimum visible duration (~400ms) once you commit to showing one. The design system records this rule; it does not ship the infrastructure — runtime flicker policy is engineering's call.

Decision tree

When you hit a wait and don't know which indicator to use, walk this. If two paths apply, the more specific wins (skeleton over spinner for content; determinate over indeterminate when count is known).

Is the wait < 300ms? Yes → no indicator No → continue Did the user just click something specific (button, row action)? Yes → inline spinner (§ Taxonomy 1 / 2) No → continue Are we populating an empty content area? Yes → skeleton (§ Taxonomy 3) No → continue Is this a route transition? Yes → render a skeleton on the destination page (the page taking shape is the feedback) No → continue Do we know the percentage / count? Yes → determinate la-progress (§ la-progress spec) No → continue Will the wait exceed 10 seconds and the user can leave? Yes → la-processing-widget (§ Taxonomy 4) No → continue Is this an AI generation? Yes → la-thinking (name locked, pattern in flight via sub-solve) No → reach for inline spinner with timeout fallback

Taxonomy

Seven loading surfaces in the product. Each entry pairs a real wait scenario with the right component (built or locked).

1 Inline action spinner Locked — slot 1

Small spinner inside a button or interactive cell — "I received your click, I'm working on it." la-button gains a loading boolean prop. Sub-solve button-loading-state next.

2 Inline cell / row spinner Informal pattern

Spinner inside a row, cell, or list item — when one item is mutating (rename, delete, reprocess) but the rest are not. Used informally via the shared spinnerStyles; no primitive yet.

3 Skeleton placeholder Locked name — slot 2

Light-grey blocks shaped like the final content. Used for the first paint of a populated region — insights list, document list, table data on tab switch. Primitive will be la-skeleton; sub-solve skeleton-primitive next.

4 Async operation widget — la-processing-widget Built

Floating, persistent widget that tracks long-running async operations (document upload, AI analysis). Survives navigation. The only loading surface that does — if the wait is >10s and the user might want to keep working, route here.

5 Stepped progress — la-stepper Built

Horizontal multi-segment bar showing position in a known-length linear flow (unlock wizard, KYC, sign-up). Display-only. Communicates position in a flow, not progress within a step.

6 Determinate progress — la-progress Locked spec

Horizontal bar that fills 0 → 100% as a known-quantity task progresses. File bytes, bulk action ("Resolving 12 of 47"), export percentage. Build-ready spec below.

7 AI thinking — la-thinking Locked name

Indicator while an AI response is being generated. Distinct from la-processing-widget — foreground and synchronous-feeling, not a background task. Sub-solve ai-thinking-state next.

la-progress — inline spec

Build-ready. No interim solve needed. Distinct from la-stepper (segmented / discrete) by name and shape.

Copy patterns

Present continuous, ellipsis at end, time estimate when known. British English, sentence case, "we" voice. Extends copy-guidelines.md.

StateCopy
Generic loading"Loading…"
Workspace bootstrap"Loading your workspace…"
Content fetch(no copy — skeleton speaks)
Document upload"Uploading your document…"
AI analysis"Analysing your document. This usually takes 10–30 seconds."
Bulk action"Resolving 12 of 47 insights…"
Export"Generating PDF — 64%"
AI thinking (synchronous)"Thinking…"
Timeout"This is taking a little longer than usual. We'll keep going."
Failure"We couldn't [verb] this. Try again." (see feedback-moments)
Don't: "Please wait" (paternalistic) · "Loading data" (vague — what data?) · "Working…" (corporate filler) · all-caps · exclamation marks.

Accessibility

The required ARIA pattern by surface. These are baseline expectations — Gate 7 of /audit.

SurfaceRequired ARIA
Spinners (button-loading, processing-widget, future thinking) role="status" + aria-label describing what is loading
Loading content regions (skeleton-wrapped) aria-busy="true" on the region while skeletons are visible
Determinate progress (la-progress) role="progressbar" + aria-valuenow / aria-valuemin="0" / aria-valuemax="100" + aria-label or aria-labelledby
Any loading state Focus must not move when the loading state appears or disappears — keep the user where they were

Anti-patterns

Concrete failure modes — each observed in adjacent SaaS products.

Spinner-in-the-void. A spinner centred on a blank page with no surrounding content. Replace with a skeleton matching the eventual layout, or a top progress bar on the populated previous page.
Double indicator. A skeleton AND a spinner. A processing-widget AND an inline button spinner for the same action. Pick one — the more specific wins.
Mismatched skeleton. A 3-row skeleton followed by a 12-row populated list. The skeleton must approximate the real layout; otherwise the page jumps and feels broken.
Eager indicator. A spinner on every action regardless of duration. A 60ms call should not flash a spinner — either no indicator or a 300ms+ delay before showing one.
Flicker indicator. A spinner that disappears in 80ms because the request returned faster than expected. Once committed, hold ~400ms minimum.
Trapped spinner. A loading state with no timeout and no failure variant. Every loading surface must have a designed failure path.
Decorative spinner. A spinner on a static surface that isn't actually loading anything ("just to look active"). Spinners are a literal claim that work is happening; lying breaks trust.
