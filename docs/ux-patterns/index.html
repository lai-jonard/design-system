UX Patterns

Standardised interaction patterns used across LawAdvisor AI. Every pattern here has shipped in at least one sprint prototype and reflects how we handle the interaction — not how we'd like to handle it hypothetically.

When a new interaction arises that doesn't fit an existing pattern, document the decision here before shipping.

1. Destructive action — Delete

When: permanently removing a document, entity, or record that cannot be undone.

How it works:

  1. Delete button is visible only when the backend confirms the item is deletable (prop deletable = true). Never show a Delete button the user cannot use.
  2. Clicking Delete opens a confirmation modal (la-modal) — never deletes immediately.
  3. The modal heading is "Delete [noun]" — e.g. "Delete document".
  4. Body copy is one sentence: "This [noun] will be permanently deleted. [Consequence sentence]." Plain, honest, no drama.
  5. Footer has two buttons: Cancel (secondary, dismisses modal) and Delete (danger variant, triggers deletion).
  6. On confirmation, navigate to the parent page (legal area page, people page, etc.). Do not leave the user on a now-empty document page.
  7. Related items (insights, linked records) update automatically on next load — copy acknowledges this: "Related insights will be updated."

Button placement:

  • Delete is always ghost-variant or secondary until it reaches the confirmation modal footer, where it becomes danger variant.
  • Delete never appears as a primary button on any surface.

Copy:

  • CTA: "Delete" — never "Remove", "Erase", or "Yes, delete"
  • Cancel: "Cancel" — never "No", "Go back", "Keep"
  • Body: "This document will be permanently deleted. Related insights will be updated."

Components used: la-modal, la-button (secondary + danger)

Prototyped in: 20-document-page.html (Delete flow), 06-corp-gov-standard.html

2. Upload / Replace flow

When: adding a new document or replacing an existing one with a corrected version.

How it works:

Replace (existing document)

  1. "Replace document" button is always shown on the document page (regardless of deletable).
  2. Clicking opens a Replace modal (la-modal wrapping la-upload). Heading: "Replace document".
  3. User drags or selects a file in la-upload.
  4. Footer: Cancel (secondary) + Upload (primary).
  5. On Upload click, modal closes, documentStatus switches to 'processing' — the existing document's About content stays visible. Replacing goes dark during processing would feel alarming.
  6. After upload completes (simulated 2s in prototype), documentStatus returns to 'ready' and aboutSummary updates to confirm the new version.
  7. If upload fails, documentStatus → 'flagged-ai' with a flagReason that explains the failure and invites retry. The original document is unchanged.

Add (new document, no existing)

  • Upload surface is shown inline on the legal area page (not in a modal).
  • After upload, document appears in the list with 'processing' status. Legal area insight count may change.

Status transitions:

idle → processing (2s simulated) → ready (success) | flagged-ai (failure)

Components used: la-modal, la-upload, la-button, la-document-page (status prop)

Prototyped in: 20-document-page.html (Replace flow), 09-fundraising-unlock.html

3. Confirmation modal

When: any action that is hard to reverse, affects records, or has downstream consequences.

Pattern:

  • Use la-modal with a single heading, one-sentence body, and a two-button footer.
  • Primary/affirmative button goes last (right side of footer). Never first.
  • Cancel always goes first (left side of footer).
  • For destructive confirmations: affirmative = danger variant.
  • For non-destructive confirmations (mark resolved, send for signature): affirmative = primary or success variant.

Footer button order (always):

[ Cancel ]  [ Affirmative action ]

Never: more than two buttons in a confirmation footer. If there are three options, one of them belongs elsewhere.

4. Insight row → Insight modal

When: user wants to understand or act on a specific insight.

How it works:

  1. Insight rows appear in the legal area page (la-legal-area-widget), the document page (la-document-page), and the dashboard (la-insight-list).
  2. Clicking any insight row fires a la-insight-select event (dispatched by the component, bubbles: true, composed: true — shadow DOM–safe).
  3. The consumer (prototype script or app) catches la-insight-select, populates la-insight-modal props, and sets modal.open = true.
  4. The modal overlays the current page — it does not navigate.
  5. Source document CTA ("View document") fires la-insight-navigate with { id: sourceDocument.id }. Consumer routes to the document page.
  6. Closing: X button, backdrop click, or Escape fires la-insight-close — consumer sets modal.open = false and modal.resolved = false.
  7. Upload action inside the modal: la-insight-upload event → consumer sets modal.uploadState = 'processing' → 2s later → modal.resolved = true.
  8. Mark-resolved action: la-insight-resolved → consumer sets modal.resolved = true.

Event chain:

click insight row → la-insight-select → open modal
click "View document" → la-insight-navigate → close modal + navigate
click upload → la-insight-upload → processing → resolved
click X / backdrop → la-insight-close → close modal

Components used: la-insight-modal, la-insight-card, la-legal-area-widget, la-document-page

Prototyped in: 06-corp-gov-standard.html, 07-insight-modal.html, 20-document-page.html

5. Two-panel document page layout

When: showing a document's details alongside its file preview.

Layout:

  • Left panel (1fr): detail — document header, About section, Linked insights, Actions.
  • Right panel (1fr): document preview — file icon placeholder (Sprint 6), PDF/DOCX viewer at integration.
  • Left panel scrolls independently; right panel is fixed within the viewport.
  • Both panels exist within a scrollable content column on the page.

Breakpoints: Desktop only in Sprint 6. Mobile/responsive deferred.

Detail panel section order (always):

  1. Document header (name, entity sub-line, status badge, meta chips)
  2. Flag banner (if status is flagged-*)
  3. About section (key-terms or obligations renderer, or loading spinner, or hidden)
  4. Details section (uploaded date, file size, category, legal area — hidden if all absent)
  5. Linked insights
  6. Actions (Replace always; Delete only when deletable = true)

Preview panel content:

  • ready / flagged-*: document-file icon + "View original" ghost button
  • processing: spinner + "Analysing document…"

Component: la-document-page

Prototyped in: 20-document-page.html

6. Legal area page layout

When: showing a legal area's health summary, insights, and document list.

Layout:

  • Full-width column within the main content shell.
  • Intro section (top): area name, status summary, unlock prompt if locked.
  • Insights widget (la-legal-area-widget): shows urgent + proactive insights grouped by severity. Clickable rows open la-insight-modal.
  • Documents list: each document is a row with name, entity, status badge, and a chevron/link to the document page.
  • Empty state: when no documents are uploaded, la-empty-state with an inviting action CTA.

Locked vs standard:

  • Locked: intro section shows locked illustration + "Start with [Area]" CTA; document list is hidden.
  • Standard (unlocked): full layout as above.
  • Unlock panel (la-unlock-panel) slides in after CTA — not a separate page; replaces the locked intro inline via state change.

Prototyped in: 06-corp-gov-standard.html, 04-corp-gov-locked.html, 05-corp-gov-unlock.html

7. Page breadcrumb navigation

When: user is on a page that is 2+ levels deep from the dashboard.

Structure (always three levels max):

Dashboard  >  [Legal Area]  >  [Document or sub-page]
  • Rendered by la-app-shell via the breadcrumbs prop (array of { label, href? }).
  • Last breadcrumb item has no href — it is the current page, not a link.
  • Dashboard is always the first item.
  • Legal area (Governance, Employment, Fundraising) is always the second item and links back to the standard legal area page.

Set in prototype:

shell.breadcrumbs = [
  { label: 'Dashboard', href: '03-dashboard.html' },
  { label: 'Governance', href: '06-corp-gov-standard.html' },
  { label: 'Employment Contract — Jane Lee' },
];

Page title (shown in the app shell header) mirrors the last breadcrumb label.

8. Empty states

When: a list or section has no content to show yet.

Rules:

  • Use la-empty-state component — never a plain <p>No items. string.
  • Always include an icon (Phosphor outline weight), a heading, a one-sentence description, and an optional CTA button.
  • Tone: inviting, actionable — "Upload your first document to get started."
  • Never: "No documents found." — passive, unhelpful.
  • CTA (if present): primary variant, verb-first — "Upload document", "Add employee".

Where they appear:

  • Document list on a legal area page (before first upload)
  • Linked insights section on document page (when no insights are linked)
  • People list before first person is added

9. Flag banners

When: a document has a status that needs attention beyond a normal insight.

Surfaces:

  • flagged-expired → danger banner (red surface)
  • flagged-unsigned → danger banner (red surface)
  • flagged-ai → warning banner (amber surface)

Layout: full-width, below the document header, above the About section. Never inside the About section.

Content: icon + title + one-sentence flagReason. If flagReason is absent, each status has a fallback:

  • flagged-expired: "This document is no longer valid and may need to be replaced."
  • flagged-unsigned: "This document requires a signature before it takes effect."
  • flagged-ai: "Our analysis flagged a potential issue with this document. Review and take action."

Component: la-document-page renders the banner internally based on documentStatus and flagReason props.

10. Processing / loading states

When: an async operation is in flight (upload, AI analysis, data fetch).

Rules:

  • Never blank the page or panel during processing. Show the previous state's content while the new state resolves. This is reassuring — users can see what was there and trust the transition is working.
  • Use a spinner (Phosphor ph-spinner-gap with CSS rotation) + short copy ("Analysing document…", "Uploading…").
  • Processing states time out or error — never leave the user in permanent spinner state. On error, surface a flag banner or inline error message with a clear action.
  • Status badge updates immediately on the component — the detail content lags until the operation completes.

Status transitions (document upload):

idle → processing (spinner in preview pane, detail content unchanged)
     → ready (new aboutSummary, status badge back to ready)
     → flagged-ai (error banner + original content restored)

11. Key–value field lists (la-field-list)

When: displaying structured metadata — document details, extracted entity facts, company information, or any set of labelled data points.

Component: la-field-list — pass a fields array of { label, value } objects.

Hierarchy through colour, not weight

Labels (left / top column) use --la-color-text-muted. Values (right / bottom column) use --la-color-text. Both sides use the same font size (--la-font-size-base) and normal weight (--la-font-weight-normal). The distinction is purely chromatic — muted label, primary-text value. Never bold or medium the value.

This mirrors the document page's general hierarchy rule: colour does the work, weight stays calm.

Layout modes

Mode Use When
spread (default) Label left, value right-aligned Narrow panels (document sidebar, ≤440px)
stack Fixed-width label column (160px), value flows right Wider surfaces (company manual, ≥480px)

Row borders (--la-color-border-light) aid scanning on longer lists. The last row's border is suppressed.

Rules

  • Hierarchy is colour only — never bold or capitalise labels.
  • Pass fields via the .fields JS property — not a string attribute.
  • Filter absent or empty fields before passing; an empty array renders nothing.
  • Use spread in the document page sidebar; use stack for company manual or entity detail pages.

12. Inline metadata strings — middle dot separator

When: displaying a horizontal run of short text labels (category, legal area, date, file size, entity name) in a single line — typically in document headers, list rows, or cards.

Rule: use a middle dot (· — U+00B7) as the separator between adjacent items. Never use a slash (/), pipe (|), dash (—), or a vertical rule element.

Implementation:

  • Render each separator as a <span class="doc-meta-sep">·</span> (or equivalent) so it can be styled independently.
  • Only render a separator when the preceding item is actually present — never render an orphaned dot at the start or end of the row.
  • Colour: --la-color-text-faint — visually lighter than the surrounding text so the dot recedes and the labels stand out.
  • Set user-select: none on the separator span so it is excluded from copy/paste.

Example:

[Acme Corp.] · Employment · 3 Mar 2024

Do not use hardcoded · character directly inside a string literal like "Employment · Governance" — always keep separators as discrete DOM nodes so conditional visibility is handled in the template, not by string manipulation.

13. Person entity modal

When: showing the details of a named person (founder, employee, advisor, or any people-entity) in response to clicking their row in a la-person-list.

This pattern covers two modal states in the same flow: the detail modal (read view) and the add modal (creation view). Both share the same header anatomy.

Header anatomy

The modal has no visible title. The person's identity is the header.

         ┌──────────────┐
         │      AC      │  ← 48 × 48 avatar circle, initials auto-derived
         └──────────────┘
          Alex Chen          ← 2xl / semibold / --la-color-text
          Founder & CEO      ← base / normal / --la-color-text-muted
  • Layout: column, align-items: center, text-align: center
  • Avatar: 48 × 48px, --la-radius-circle, --la-color-bg-active fill, --la-color-text-secondary text, font-size-base, semibold
  • Name: --la-font-size-2xl, --la-font-weight-semibold, --la-color-text
  • Subtitle (role / title): --la-font-size-base, normal weight, --la-color-text-muted
  • Gap between avatar and text stack: --la-space-sm
  • Bottom margin before body content: --la-space-xl

No modal heading (heading="") — the visual title is redundant. Set aria-label on <la-modal> to the person's name for screen readers.

modal.setAttribute('aria-label', person.name);

Screen readers announce the dialog name on open, then read all body content sequentially. The visible title bar is removed entirely.

Detail modal body

Below the header, show the person's associated content. The exact sections depend on context:

  • Founders / Governance: documents list (la-document-item rows) with a normal-weight, muted section label ("Documents") indented to align with document item content (padding-inline-start: --la-space-md)
  • Employment / People: metadata field grid (start date, contract type, etc.) + contract link

Keep section labels at --la-font-size-sm, --la-font-weight-normal, --la-color-text-muted. Never semibold — the label should recede, not compete with the content.

Add modal

Same header anatomy, but the name and subtitle start as placeholder text:

  • Name slot shows "Enter a name below" in --la-color-text-disabled — most receded text token
  • Subtitle slot shows the role statically (e.g. "Founder", "Employee") — always visible, never hidden
  • As the user types, initials update live in the avatar and name slot updates to the typed value in --la-color-text
  • Avatar shows — (em dash) until a name is entered

The la-input field sits below the header, inside the modal body.

Footer — detail modal

Destructive action is isolated on the left. Close is on the right.

[ Remove [noun] ]                    [ Close ]
  ghost + --la-color-danger-text       ghost
  margin-right: auto
  • "Remove" trigger uses variant="ghost" with color: var(--la-color-danger-text) — visible as a caution signal without the weight of a filled danger button
  • The filled danger variant is reserved for the confirmation modal (the final destructive step), not the trigger

Use display: contents on the footer <div> so buttons become direct flex children of the modal footer and margin-right: auto on the left button correctly pushes it to the far left.

<div slot="footer" style="display:contents;">
  <la-button variant="ghost" style="margin-right:auto;color:var(--la-color-danger-text);">
    Remove founder
  </la-button>
  <la-button variant="ghost">Close</la-button>
</div>

Footer — add modal

Standard two-button footer, right-aligned (modal default):

                         [ Cancel ]  [ Add [noun] ]
                           ghost       primary, disabled until name entered

Footer — confirmation modal

Destructive confirm is on the left. Cancel is on the right.

[ Remove ]                           [ Cancel ]
  danger variant                       ghost
  margin-right: auto

Triggering the flow

click la-person-item row
  → fire la-select event ({ name, subtitle })
  → populate and open detail modal

click "Remove [noun]" in detail modal
  → close detail modal
  → open confirmation modal with person's name

click "Remove" in confirmation modal
  → remove person from data array
  → re-render la-person-list
  → close confirmation modal

Components used: la-modal, la-person-list, la-person-item, la-button, la-input, la-menu + la-menu-item (kebab on person row)

Prototyped in: 13-company-manual.html (founders), 18-people.html (employment)

14. Scrollable content — fade to background

Rule: Any contained, bounded scroll area uses a gradient fade at its trailing edge to signal that content continues beyond the visible boundary. No hard border or divider line.

Rationale: A border says "here is a wall." A gradient fade says "there is more." Borders add visual weight and work against the calm, low-noise aesthetic. The gradient is subtle by design — it disappears entirely when there's nothing more to show.

When it applies

Context Trailing edge
Modal body overflowing (la-modal, la-insight-modal) Bottom of body, above footer
Drawer body overflowing (la-drawer) Bottom of body, above footer
Select dropdown overflowing (la-select) Bottom of panel
Any future bounded scroll region Bottom (or right for horizontal)

Does not apply to:

  • Full-page scroll containers (la-app-shell main, document pages) — the page edge is the natural boundary
  • Horizontal data table overflow (la-data-table) — handled separately
  • Sidebars with distinct background colours — gradient colour would need to match the surface

Implementation — with a footer sibling

When the scrollable area has a static footer below it (la-modal, la-drawer, la-insight-modal):

.footer {
  position: relative; /* establishes containing block for ::before */
}
.footer::before {
  content: '';
  position: absolute;
  bottom: 100%;   /* sits above the footer, overlapping the body */
  left: 0;
  right: 0;
  height: var(--la-space-2xl);   /* 32px */
  background: linear-gradient(to bottom, transparent, var(--la-color-bg));
  pointer-events: none;
}

For la-modal the gradient is gated behind [scrollable] (JS detects body overflow). For la-drawer it is always present — drawers consistently contain substantial content.

Implementation — self-contained scroll panel (no footer)

When the scrollable element has no footer sibling (la-select), add a <div class="scroll-fade"> as the last child of the panel in the shadow template:

.scroll-fade {
  display: none;
  position: sticky;
  bottom: calc(-1 * <panel-padding>);
  height: var(--la-space-2xl);
  margin-top: calc(-1 * var(--la-space-2xl));
  background: linear-gradient(to bottom, transparent, var(--la-color-bg));
  pointer-events: none;
}
.panel[data-scrollable] .scroll-fade { display: block; }

JS sets data-scrollable when panel.scrollHeight > panel.clientHeight, checked on open and on scroll.

Gradient value

Always fade to var(--la-color-bg) (white). Components on non-white backgrounds require a matching target colour — handle case by case.

Pattern 15 — Three-state access pill (sharing status)

Used on the investor detail page header and investor list cards to communicate whether a DD pack has been shared with a specific investor. A composed pattern using la-badge + la-tooltip.

Three states

State Pill Tooltip
Not Shared neutral — "Not Shared" "No invite sent yet. Use Share DD Pack to give this investor access."
Active success + ph-fill ph-check-circle icon — "Active" "Access is live — this investor can view the DD pack."
Revoked warning + ph-fill ph-prohibit icon — "Revoked" "Access has been revoked. The investor can no longer view the pack."

Status mapping

The three pill states map from the more granular engagement status used internally:

  • Not Shared → draft, invited
  • Active → active, reviewing, stalled, complete
  • Revoked → access-revoked

Usage

<la-tooltip tip="Access is live — this investor can view the DD pack."
            appearance="light"
            position="bottom"
            max-width="220px">
  <la-badge variant="success">
    <i class="ph-fill ph-check-circle" slot="icon"></i>Active
  </la-badge>
</la-tooltip>

Always use appearance="light" — this is supplementary context on an already-labelled element, not an action label. Do not use size="sm". Use max-width="220px" to allow longer tip text to wrap cleanly.

Contextual CTA alongside the pill

The pill always sits next to a single contextual CTA in the header investor-header-end slot. The CTA is determined by the pill state:

  • Not Shared → "Share DD Pack" (primary button)
  • Active → "Send nudge" (secondary) + Revoke in la-menu kebab
  • Revoked → "Re-share" (secondary button)
