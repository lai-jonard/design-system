Playbook — How the LAI Design System grows

This is the recipe for how new ideas enter the design system, how they become components, how decisions get logged, and how work hands off to engineering. It's the single page a new collaborator reads to understand the workflow — sitting alongside CLAUDE.md (the boiled-down non-negotiables: token, copy, motion, code), the rest of design-language/ (the rationale layer with worked-out detail — components.md for taxonomy, copy-guidelines.md for tone and voice, motion.md for animation), and handoffs/README.md (the engineering pipeline).

If you're picking up a new design task and you don't know whether to solve, build, or just sketch — start here.

1. The mental model

Six layers, each with a clear job:

Layer Lives in Job
Tokens src/styles/tokens.ts Colour, radius, spacing, type, motion. Single source of truth. Hardcoded values are a rule violation.
Primitives src/primitives/ Reusable building blocks (button, input, badge, drawer). No domain knowledge.
Composed components src/components/ Domain-aware compositions (insight-card, unlock-panel, document-list). Each entry is in the Tier 2 table in components.md with a logged "what problem does this solve".
Page templates src/pages/ Layout shells (app-shell, dashboard, legal-area-page). The structural canvases the product surfaces sit on.
Prototypes dev/prototypes/<flow>/ Flow-level explorations — full user journeys assembled from primitives and components. Promoted to published references via the 8-gate prototype standard.
Flows (logical, not a folder) A user journey across pages: onboarding, unlock, document review. Prototypes test these.

The boundary that matters most is primitive vs composed component. Primitives have no domain knowledge — la-badge doesn't know it's a "legal area badge". Composed components do — la-legal-area-widget knows it counts urgent insights. Cross that line carefully: a composed component should never end up reused outside its domain, but a primitive should be reusable across products (KYC, sign-up, future verticals).

2. The recipe (one-page, scannable)

A new product need or design question appears. Walk this:

1. Is this a question or a settled call?
   • Question with multiple valid answers → /solve
   • Settled call needing a record → write a decision (see §3)
   • Settled call ready to build → skip to step 4

2. /solve framed the problem, output a brief.
   Did the brief settle the question?
   • Yes → continue to step 3
   • No → keep the solve open, refine

3. Does it need new UI?
   • New visual pattern, no component → log under "Known divergences" in
     design-language/README.md or in the relevant area .md
   • New component → apply the rubric (§4), log Tier 2 entry in components.md

4. /build the component.
   Scout (read-only) → align (propose plan, wait) → build under constraints.
   Two stops, no scope creep.

5. /audit at any time to inventory state.
   Passive report — no edits beyond readiness.md row.

6. /document when the API is settled.
   Drafts dev/docs-drafts/la-<name>.html. Iterate.

7. /ready when the component is finished.
   Walks 9 gates, promotes draft to docs-src/, updates catalogue chip,
   flips readiness, prompts for new rules and decisions, updates sprint
   handoff. This is the last mile.

8. Decision recorded in decisions/NN-*.md (always append, never edit).

9. Sprint handoff bundle in handoffs/stage-N/<sprint>.md updated.
   Engineering builds from there.

Most components don't need a solve. Most solves don't need to spawn new components. Most decisions don't need their own file. Use the lightest tool that works.

3. When to solve, decide, or just build

Three patterns, three different tools. Pick by the shape of the question, not by habit.

Tool When What it produces What it costs
/solve The question has multiple valid answers and we need to think it through. A solve workspace at solves/YYMMDD-N-slug/ with three files: *.solve.md (problem + scope), *.solve-log.md (live journal), *.solve-output.md (distilled brief). A session of focused thinking. Don't solve trivial calls.
decision The call is made, but it'll bind future work and someone will need to know why. A new file decisions/NN-<slug>.md, append-only. Picks up the next number. Five minutes of writing. Cheap.
/build The answer is settled and there's a component to make. A new or modified component in src/. Run /document and /ready after. A real implementation pass.

Rules of thumb:

  • If you're tempted to solve a question that's already answered elsewhere, look first — decisions/, solves/, design-language/ may already have it.
  • If you're about to write a decision that contradicts an earlier one, the new entry should link the superseded one. Never edit closed decisions.
  • If a /build exposes a question you weren't expecting, stop, open a /solve. Don't keep building through ambiguity.

4. The component decision rubric

Every Tier 2 entry in components.md starts with the question "What problem is this component solving?" — but that's the prompt, not the rubric. Use these criteria to judge whether the answer warrants a component:

A pattern becomes a component when two or more of the following are true:

  1. Used in 2+ contexts — already appears, or will appear, in more than one consumer.
  2. Distinct visual shape — has its own identifiable anatomy, not just a styled <div>.
  3. Owns a state machine — has internal state (open/closed, loading, selected, etc.) or behaviour worth encapsulating.
  4. Reusable across products — plausibly useful in a future vertical (KYC, sign-up, investor flow).

If only one is true, inline it in the consumer. Premature componentisation creates ghost APIs that no one uses. If we later find the inlined pattern reappearing, we promote it then — that's cheaper than maintaining a never-reused primitive.

Edge cases:

  • One consumer, distinct shape, no state machine → inline. (Example: a one-off banner specific to a single onboarding step.)
  • Two consumers, identical visual, no state → token + utility class is enough; no component needed.
  • One consumer today, planned second consumer this quarter → component. Plan beats inertia.

5. The exploration → component pipeline

Prototypes are how we test ideas at flow-scale before deciding what becomes a component. They live in dev/prototypes/<flow>/<NN>-<name>.html and assemble real primitives and composed components against realistic content.

The promotion gate is the 8-gate prototype standard (prototype-standards.md):

  1. Source — file exists at the canonical path
  2. Components — every <la-*> tag is published
  3. Design language — novel patterns are documented or justified
  4. Behaviour — motion via motion.css, GPU-safe, keyboard navigable
  5. Responsive — reflows clean at 1440 / 1024 / 768
  6. Content resilience — realistic sample data, no Lorem ipsum
  7. Accessibility — WCAG AA, keyboard, focus, ARIA, colour-plus-something
  8. Standards compliance — tokens only, asset paths absolute, copy guidelines

A prototype that passes all eight is a published reference — it can be cited from sprint handoffs and engineering can lean on it. A prototype that doesn't pass stays in dev/prototypes/ as exploration only.

Lessons from prototypes feed the Tier 2 table in components.md. When a prototype repeatedly reaches for the same shape (a checklist, a teaser banner, an empty-state surface), that's the signal to promote the shape to a composed component via the rubric in §4.

Note: this is a separate "8 gates" from the 9 readiness gates in /ready. Don't mix them up. Prototype gates are about the prototype as a whole; readiness gates are about a component's docs and state coverage.

6. Slash-commands reference (designer POV)

The engineer set up five custom commands under .claude/commands/ that automate the workflow. This is what each does for me, the designer. (Engineer to QA the technical detail of this section.)

/solve <topic> — frame a question

Opens a solve workspace at solves/YYMMDD-N-slug/ with three files. Used when a design question has multiple valid answers and we want to think through the boundaries.

  • Frames the problem rigorously rather than jumping to solutions.
  • Captures constraints, principles (with their because), open questions, references.
  • Verifies framing by handing the solve file to a sub-agent with no other context — if the sub-agent can produce sensible output, framing is sufficient. If not, the solve file has gaps.
  • Use when in doubt. Cheap to open, cheaper than building wrong.

The output (*.solve-output.md) is the distilled brief that becomes the input to /build, or the source for a decisions/ entry.

/build <name> — scout, align, then build

Builds a new component or modifies an existing one. Three hard stops:

  1. After scout — present findings, wait for acknowledgement
  2. After alignment proposal — wait for go-ahead
  3. After build — suggest /document, do not self-declare done

Builds the component source and routes straight to /document — there's no separate sandbox step.

Reads real tokens from tokens.ts — won't carry over invented token names from a prompt. Refuses to introduce new primitives without explicit approval. Does not touch handoff artefacts; that's /ready's job.

/audit <name> — passive state report

Inventories an existing component against the readiness gates. Reports only — does not fix. Updates the row in handoffs/readiness.md, nothing else.

Use it when:

  • A legacy component pre-dates the workflow and we want to know where it stands.
  • Something looks off and we want a quick gap report before deciding whether to /ready.
  • Periodic sweeps (e.g. cycle-end) to catch drift.

The report orders gaps easiest-first, so we know what's quick wins vs. deeper work.

/document <name> — draft the docs page

Drafts the public-facing documentation page at dev/docs-drafts/la-<name>.html. Stays in dev/ so we can iterate via npm run dev without touching the published surface.

  • Pulls API truth from the component source (every @property, <slot>, dispatchEvent).
  • Pulls control structure from the component itself.
  • Pulls curated copy from design-language/ — never invents copy.
  • Anatomy section mirrors the render tree; placeholder regions get flagged, not invented.

Drafts are not published. /ready promotes them to docs-src/components/.

/ready <name> — promote, lock, hand off

The last mile. Walks nine gates (the 8 from /audit plus Gate 9: docs draft exists), and only when all pass does it:

  1. Promote dev/docs-drafts/<name>.html → docs-src/components/<name>.html (Eleventy template format).
  2. Delete the draft.
  3. Update the catalogue chip in docs-src/components/index.html.
  4. Flip the row in readiness.md to 🟢.
  5. Prompt for any new design-language rules or decisions to log — appends to design-language/*.md or creates decisions/NN-*.md.
  6. Update the active sprint handoff in handoffs/stage-N-<name>/.

The self-audit step is strict: every claim must trace to a file actually read in this session. A single yellow gate keeps the status at 🟡, not 🟢.

If Gate 9 fails (no draft exists), the command refuses to flip to 🟢. Run /document first.

7. Handoff to engineering

Two scopes — per-component and per-sprint.

Per-component: when /ready flips a component to 🟢, the handoff artefacts are emitted automatically:

  • Published docs at docs-src/components/la-<name>.html (rendered by npm run docs).
  • Updated row in handoffs/readiness.md with notes on what's in / out / where catches exist.
  • Any new rule appended to a design-language/*.md file.
  • Any new decision in decisions/NN-*.md.
  • Updated sprint matrix in handoffs/stage-N/<sprint>.md.

Per-sprint: each sprint folder under handoffs/stage-N/ is a thin wrapper that points at real artefacts (solves, decisions, prototypes, ready components). The sprint doc itself contains:

  • Scope and goal
  • Component readiness matrix
  • Edge cases answered
  • Open questions log
  • Changelog of what landed

Engineering reads the sprint doc, follows the pointers, and builds. The handoff call is a checkpoint, not a transmission — by the time we sit down, the bundle should already answer most questions.

For the broader pipeline (roles, who writes what), see handoffs/README.md.

Two surfaces don't exist yet but should:

  • solves/README.md — onboarding for the solve workflow. When to open one, what each of the three files is for, when to close.
  • decisions/_template.md — concrete template so new entries follow a consistent shape. (Forty-one decisions exist with informal but consistent format; codifying it removes friction.)

Both are tracked in the Cycle 4 plan.

9. When things break the recipe

Edge cases the recipe doesn't cover cleanly. When you hit one, treat the recipe as the default and the deviation as a deliberate, logged choice.

A pattern doesn't fit Tier 1 or Tier 2. Open a solve. Don't force-fit. Sometimes the answer is "this isn't a component, it's a layout" or "this belongs in the page template, not as its own composition."

A new decision contradicts an earlier one. Write a fresh decisions/NN-*.md and link the superseded entry at the top of the new file. Never edit a closed decision — the historical record matters. The most recent decision wins; older ones note "superseded by NN".

A 🟢 component needs to evolve. Convention: flip the readiness back to 🟡 with a note pointing at the change, log a fresh decision record explaining what's changing and why, then run /build (modify) → /document → /ready to relock at 🟢. Don't silently mutate a 🟢 component — engineering may already be building against it.

A solve produces sub-questions that need their own solves. Open them as separate solve workspaces with their own YYMMDD-N prefix. Cross-link in both directions. Keeping solves narrow keeps them tractable.

A prototype passes seven of the eight gates. That's still a prototype, not a published reference. Either close the gap or stop calling it published. Half-published is a future regression vector.

Engineering finds an inconsistency at handoff. That's why the handoff call exists. Capture in handoffs/stage-N/questions-log.md, then either resolve into a fresh decision record (if it's a call we own) or fold back into a solve (if it's a question we hadn't seen).

10. Lessons feeding back

The recipe should improve over time, not stay frozen. Two mechanisms:

Per-sprint retro. At the end of each sprint, append a short "if I ran this again" section to handoffs/stage-N/<sprint>.md. One paragraph. What surprised us, what we'd cut, what we'd surface earlier. These are the raw material for the next playbook revision.

Quarterly playbook review. Read this file end-to-end against what we actually did the last three months. Anywhere the recipe diverged from reality, either update the recipe or flag the divergence as a known exception. The playbook should never be more idealised than the practice.

Designer-engineer pairing checkpoint. Schedule a recurring half-hour with the engineer who owns the slash-commands and infrastructure. The recipe goes stale fastest at the seams (where designer process meets engineer tooling) — the pairing slot keeps both sides current with the other's reality.

Visual regression sweep (light). Once per cycle, walk the published docs pages of every 🟢 component and screenshot them against the previous cycle. Drift catches you faster than a full Chromatic-style infra and is cheap enough to keep running.

Quick reference

If you remember nothing else from this file:

  • Tokens before pixels. Every visual value is a CSS variable. No exceptions.
  • Solve before build when the question is open. Build before solve when the answer is clear.
  • Two yes on the rubric before a pattern becomes a component.
  • Decisions are append-only, numbered, never edited after closure.
  • Readiness flips happen only via /ready, never by hand-editing readiness.md.
  • Drafts live in dev/, published lives in docs-src/. /ready is the only thing that crosses the line.

When the recipe contradicts what feels right, the recipe is the default but not the rule. Diverge deliberately, log the divergence.
