design-language/interaction.md

Interaction

Buttons (la-button)

Variants

Variant Background Text Use
primary --la-color-primary white The main action on a screen. One per view.
secondary white --la-color-text Cancel, back, or secondary actions
success green tint green Approve, confirm, complete
danger red tint red Destructive — delete, remove, reject
ghost transparent --la-color-text Low-prominence links or tertiary actions

Sizes

  • Default: font-size: var(--la-font-size-md), padding: 6px 16px
  • sm: use in compact contexts like table rows or tight panels

Disabled state

opacity: 0.4 on the element — no separate disabled token. Preserves variant identity so a disabled danger button still reads as danger.

CTA copy rules

  • Verb-first, Title Case: "Send for Signature", "Add Employee", "View Details", "Mark Resolved"
  • No jargon: "send" not "submit", "check" not "verify", "start" not "commence"
  • One primary CTA per view — never show two primary-variant buttons simultaneously. Use secondary or ghost for additional actions.
  • Full capitalisation rule: see design-language/copy-guidelines.md — Capitalisation.

Badges (la-badge)

Used inline to communicate status or category. Always sentence case, 1–2 words max.

Variant Meaning
danger Urgent — action required
warning Proactive — should be addressed
success Resolved / positive state
neutral Locked, inactive, unknown
info Processing, in-progress
accent Category label (not a status)
primary Prominent label / default counter colour

Counter type (circle) is a separate shape used for numerical counts — unread items, queue depth, notification totals. Use appearance="ghost" when the count should sit quietly in the UI; default (filled) for primary signals.

Checklist items (la-checklist-item)

Presentational only — not a form input. Shows checked/unchecked state for onboarding and unlock flows. Does not emit events.

Progress steps (la-stepper)

Multi-step flows use numbered step indicators. Each step has a status:

Status Visual Meaning
pending Empty circle Not yet reached
active Highlighted Current step
complete Filled Done
processing Animated In progress, awaiting result
skipped Muted Deliberately bypassed
failed Error colour Needs attention

Hover states

  • Primary: darkens to --la-color-primary-hover
  • Secondary: light grey background
  • Success/Danger: switch to semantic surface + semantic border colour on hover (signals intent before the user commits)

Rule: never remove a border on hover. Border changes on hover are semantic signals — removing them causes layout shift and loses meaning.

Focus states

Inputs and interactive controls use --la-color-border-focus (#3B82F6) as the focus ring. Never suppress :focus-visible outlines.

Optical alignment — icon + text buttons

Geometric centring makes a trailing/leading icon look pushed too far to its edge. Give the icon side ≈2px less padding than the text side so the button reads as balanced:

/* trailing icon: "Send for Signature →" */
padding-left: 16px;
padding-right: 14px;  /* icon side = text side − 2px */

For a lopsided glyph (caret, star, play triangle), fix the SVG's viewBox rather than nudging with margin where you can. See decision 83.

Minimum hit area

Interactive controls target a 32×32px minimum hit area for our dense desktop context. Where the visible glyph is smaller (checkbox, ph-dots-three menu, close button), extend the hit area with a pseudo-element rather than enlarging the glyph:

.control { position: relative; }
.control::after {
  content: ""; position: absolute; inset: 50% auto auto 50%;
  width: 32px; height: 32px; transform: translate(-50%, -50%);
}

32px clears the WCAG 2.5.8 (AA) 24px floor. Two interactive elements must never have overlapping hit areas — shrink the pseudo-element before it collides. Revisit toward 40/44px if a touch or mobile surface becomes a real target. See decision 83.

Widgets (la-widget)

  • highlighted — primary colour border, use for recommended or active widgets
  • clickable — pointer cursor, use when the whole widget is interactive
  • dimmed — reduced opacity, use for locked or unavailable states

Modals

Opening

  1. Set backdrop display: flex
  2. Apply animation: fade-in to backdrop
  3. Remove → reflow → re-add .anim-scale-in on inner panel

Closing

  • X button (top right)
  • Cancel button (footer)
  • Click backdrop
  • Escape key

Multi-step modals

Replace innerHTML of the inner panel rather than stacking separate modals. Re-trigger .anim-scale-in on each step so the user feels forward progress.
