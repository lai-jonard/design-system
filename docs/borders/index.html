Borders

The two-token system

Borders in this system are split into two categories with separate tokens. This is one of the most important layout rules.

Token Value Category
--la-color-border #DADDEC Interactive — UI controls
--la-color-border-light #ECEEF5 Structural — page layout

--la-color-border — interactive components

Used on anything the user directly interacts with or focuses on. The slightly darker value ensures these borders remain legible against surface backgrounds and meet contrast expectations for form controls.

Use on:

  • Buttons (secondary, danger ghost, success ghost)
  • Text inputs, selects, checkboxes
  • Modals and drawers (outer ring)
  • Toast notifications
  • Cards with clickable or hoverable states
  • Dropdown menus

--la-color-border-light — structural dividers

Used on chrome and structural elements that divide areas of the layout. These borders should recede — they frame content without competing with it.

Use on:

  • Table row dividers (th bottom, td bottom)
  • Panel left/right edges
  • Section dividers within panels (border-bottom on headers)
  • Step strips and breadbars
  • Page-level horizontal rules

Why a separate lighter token: the table row borders were the original pain point — #DADDEC on every row felt visually noisy, especially in dense contract tables. Dropping to #ECEEF5 lets the data breathe. The principle extends to all structural chrome: borders that exist only to separate regions should be quieter than borders on things you interact with.

Semantic borders

These pair with their corresponding surface and action colours. Used in context-specific states only.

Token Value Use
--la-color-border-success #32A66E Success hover on ghost buttons, success alert outlines
--la-color-border-danger #CC4643 Danger hover on ghost buttons, danger alert outlines
--la-color-border-warning #F59E0B Warning notification bars, warning alert outlines
--la-color-border-focus #3B82F6 Input focus ring
--la-color-border-active #18181B Active/selected outline — sidebar rail, active tab, selected card

Hover border rule for ghost buttons

On hover, ghost buttons (danger, success) switch their border from --la-color-border to the semantic border colour. This signals intent before action:

.btn-danger-ghost:hover  { background: var(--la-color-surface-danger);  border-color: var(--la-color-border-danger); }
.btn-success-ghost:hover { background: var(--la-color-surface-success); border-color: var(--la-color-border-success); }

The border is always kept on hover — never removed. Removing borders on hover creates a layout shift and makes the button feel unstable.

Border width

  • 1px — standard for all borders
  • 1.5px — table header bottom rule only (slightly heavier to mark the boundary between header and data)
  • Dashed borders — reserved for drop / upload / add targets only (see below). Never for structural chrome, content cards, controls, or passive empty states.

When to use a dashed border

Dashed = a surface that accepts something — a drop zone, upload slot, or "add" target. The test: can I drop or add something into this box right now? If yes → dashed. If it's just empty, or it's real content / chrome → solid.

Surface Border
Upload / drop zone, "Add X" slot, signature-placement spot Dashed
Passive empty state ("No deals yet", "no items yet") Solid hairline (--la-color-border-light), optionally on a subtle tint — calm, not provisional
Content card, populated row Solid --la-color-border
Structural divider / chrome Solid --la-color-border-light
Buttons, inputs, tabs Solid --la-color-border

States of a drop target:

  • Resting — dashed, --la-color-border.
  • Active (hover / drag-over) — border → --la-color-border-active, faint tinted fill ("release here").
  • Filled — flips to a solid border; the dash only ever signalled "empty, drop here".

This mirrors the mainstream convention (Figma/Drive/Trello drop targets are dashed; Linear/Stripe render passive empty states as solid hairlines or tinted panels, reserving the dash for add/drop).

Dashed border — standard treatment (rendering)

CSS border: dashed can't control dash length, gap, cap style, or stroke alignment, so the standard dashed border is rendered as an SVG rect overlay:

  • Dash 3 · gap 5 · round dashcaps · 1px centred stroke, colour --la-color-border.
  • Corner radius follows the host shape — the overlay's rx/ry read a --dash-radius variable, which the host sets to the same token it uses for its own border-radius, so the dashes always trace the element's corners.
  • The stroke is non-scaling (vector-effect="non-scaling-stroke"), so the 1px width and the 3/5 rhythm stay constant on responsive/full-width elements.

Reusable utility: dev/prototypes/dash-border.css (class .dash-border). Tunable per host via --dash-len / --dash-gap / --dash-width / --dash-color / --dash-radius; add .is-active for the drag-over state. Usage:

<!-- host: position: relative; --dash-radius: var(--la-radius-lg); border-radius: var(--dash-radius); -->
<svg class="dash-border" preserveAspectRatio="none" aria-hidden="true">
  <rect x="0" y="0" width="100%" height="100%" vector-effect="non-scaling-stroke"></rect>
</svg>

When the dashed border moves into a component (e.g. la-upload drop zone), bake the same SVG-overlay treatment in rather than reverting to border: dashed.

No border on filled buttons

Primary, neutral, and dark buttons use border-color matching their background, making the border invisible. This is intentional — filled buttons do not need a visible border to feel solid. The border exists in the DOM only so that button sizes remain consistent when switching between filled and outlined variants.

Concentric radius on nested surfaces

When a rounded element sits inside another rounded element, the corners must stay concentric — parallel curves, not a tight corner fighting a loose one. Mismatched nested radii are the single most common thing that makes a layout feel subtly off.

Rule: outer radius = inner radius + padding between them.

/* card padding: --la-space-sm (8px), inner button radius: --la-radius-sm */
card    → border-radius: calc(var(--la-radius-sm) + var(--la-space-sm));
button  → border-radius: var(--la-radius-sm);

Applies when the surfaces are close together. Once the padding is large (≳24px) the layers read as separate surfaces — pick each radius independently from the scale rather than forcing the maths. Check this on any card-with-inner-control: la-insight-card, la-panel, la-offer-banner.
