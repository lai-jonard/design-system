Layout

What this library provides

This is a component library, not an app shell. Layout components (sidebar, page header, tabs) are not part of this library — they live in the consuming application. What this library provides are self-contained components that work inside any layout.

These principles describe how screens and flows built with this library should be structured.

Content max-width

900px, centred with margin: 0 auto (or margin-inline: auto).

Applies to the main content column of every page — dashboards, legal area pages, list/document pages. The aside in la-app-shell is a separate slot and sits outside this constraint. 900px keeps line lengths comfortable, works for both card grids and tables, and is consistent across all content types.

The 900px standard was settled during Sprint 2 dashboard work (2026-05-06) after reviewing actual widths across all prototype pages. Previous docs said 1280px — that was aspirational and never matched the components.

Spacing scale

Context Value
Content area padding (top) 24px
Content area padding (horizontal) 28px
Gap between cards / feed items 14px
Inner card padding 16–20px
Modal padding 28px
List item padding 8px 10px

Two-panel layout

For screens where a user works with a document or detail view alongside a control panel (e.g. review, negotiation, unlock flow).

[container]         display: flex
├── [left panel]    fixed width ~380–580px, scrollable
└── [right panel]   flex: 1, scrollable
  • Left panel: actions, steps, contextual controls
  • Right panel: the primary content (document, detail, preview)
  • Divider: border-right: 1px solid var(--la-color-border-light)

Card-based layout

The default layout pattern for dashboard and area pages. Cards sit on a page background (--la-color-bg-subtle) and are the primary content surface.

  • Specialised cards (e.g. la-insight-card, la-investor-card) own their own chrome. Use la-card as a generic fallback for ad-hoc titled sections, or la-panel for de-emphasised content that shouldn't float
  • Grid: display: grid; grid-template-columns: repeat(auto-fill, minmax(300px, 1fr)); gap: 14px
  • Full-width components (lists, panels) use grid-column: 1 / -1

Modals

  • Max-width: 560px (standard), 640px (form/intake modals)
  • Border-radius: --la-radius-xl2 (24px)
  • Padding: 28px
  • Backdrop: --la-color-scrim (rgba(0, 0, 0, 0.2)) — shared across modals, sheets, and any UI that dims the page behind a surface
  • Animation: backdrop fades in, inner panel scales in (.anim-scale-in from motion.css)
  • Max-height: 90vh with overflow-y: auto
  • Close: X button, cancel button, click backdrop, or Escape key

Component placement principles

  • White space first — components are never crowded. Give them room.
  • No coloured section backgrounds — surfaces are white or near-white only
  • Borders are structural, not decorative — use --la-color-border-light for layout dividers, --la-color-border for interactive components
  • Shadows only on floating elements — modals, dropdowns, toasts

Responsive breakpoints

Name Width Context
Mobile < 768px Single-column layout, stacked cards
Tablet 768px - 1023px Simplified grid, collapsible sidebar
Desktop >= 1024px Full layout with sidebar and multi-column grids

Grid patterns

  • Use auto-fill / auto-fit with minmax() for card grids — never fixed column counts.
  • Cards reflow naturally as the viewport shrinks.
display: grid;
grid-template-columns: repeat(auto-fill, minmax(300px, 1fr));
gap: 14px;

Layout shifts

  • Dashboard: flex row on desktop, column stack at 768px
  • Two-column layouts (e.g. data room): grid-template-columns: 1fr 1fr on desktop, single column below 800px
  • Sidebar: always visible on desktop, collapsible on tablet, overlay on mobile

Aside panel at tablet widths (768–1023px)

The la-app-shell aside (fixed-width right panel containing actions and documents) squeezes the main content uncomfortably at tablet. Evaluated options:

Option How it works Best when Trade-off
1. Toggle collapse Button in header shows/hides aside. Hidden by default at ≤1023px. showAside prop already exists. Aside is supplementary — occasional reference Aside is hidden by default; user must discover it
2. Stack below .body switches to flex-direction: column at tablet; aside gets width: 100% Read-first workflow — user reads content, then scrolls to act Actions feel disconnected from content; more scrolling
3. Overlay drawer Aside slides in as a full-height overlay panel from the right. Button in header triggers it. Backdrop behind. Users switch frequently between content and actions but not simultaneously Requires a drawer component (not yet in system); content obscured when open
4. Tab bar .body becomes a two-tab layout at tablet: "Content" tab and "Actions & Documents" tab Distinct workflows — read vs. act, not used side by side No simultaneous view; switching has friction
5. Condensed strip Aside narrows to ~56px icon-only at tablet (mirrors sidebar collapsed state) Pure action panels with few discrete actions Document lists don't translate to icon-only

Decision: overlay drawer (Option 3).

Confirmed after thinking through the founder workflow:

  • The main content shows insights — read-only, consumed not edited. Founders spend time reading and understanding here.
  • The aside is intake — adding context, data, and documents that improve insight quality.
  • These are two distinct modes: consume → act. Founders context-switch cleanly between them, not cross-reference simultaneously.
  • An overlay gives full-width content when reading, and full-attention aside when acting. The squeeze problem doesn't exist because the modes don't coexist.
  • On desktop the aside is always visible as ambient context. On tablet, forcing mode-switching is the right constraint, not a compromise.

Future direction: The aside will grow to include step-by-step plans and guided actions. This reinforces the overlay model — it's a focused coaching workflow, not a reference panel.

Trigger design — this is where the delight lives: The entry point into the overlay should not be a generic "open panel" toggle. It should frame the value:

  • "3 ways to strengthen this insight" — founders understand why they're opening it before they do
  • Positive framing — not a chore, a win
  • Placement: persistent CTA in the page header ("Improve this insight"), not a hidden toggle

Aside framing principle — coach, not form: Even though the aside is collecting data, it should never feel like a form. Each item should carry a why: "We're missing your funding history — adding it would sharpen this insight significantly." Progress indicators and completion states reinforce the behaviour. Empty state: inviting and explanatory, not blank.

Touch targets

Minimum 44px for all interactive elements on touch devices. No horizontal scroll — content must fit viewport width.

Test widths

Always check at: 375px (mobile), 768px (tablet), 1280px (desktop).
