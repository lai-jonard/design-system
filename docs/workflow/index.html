How we design

The design workflow

How ideas become interface at LawAdvisor AI — from a single design token to a full product journey, and how that work reaches engineering. The system is built for calm authority: every visual value is defined once, every non-obvious call is written down, and nothing ships without passing the same gates.

Six layers, one system

Everything in the library sits in one of six layers. Each builds on the one below it — so a change to a token ripples predictably all the way up to the screens people use.

1
Tokens src/styles/tokens.ts
Colour, type, spacing, radius, motion — defined once. The single source of truth. Nothing hardcodes a value.
2
Primitives src/primitives/
Reusable building blocks — button, input, badge, drawer. No domain knowledge; usable across any product.
3
Composed components src/components/
Domain-aware compositions — insight card, unlock panel, document list. Each answers a logged "what problem does this solve".
4
Page templates src/pages/
The structural canvases — app shell, dashboard, legal-area page — that product surfaces sit on.
5
Prototypes dev/prototypes/
Full journeys assembled from real components against realistic content — where ideas are tested before they harden.
6
Flows
A journey across pages — onboarding, unlock, document review. The thing a founder or investor actually experiences.

How an idea becomes interface

A new product need walks the same path every time. Each step has a clear owner and a clear output, and nothing skips a gate.

Solve
Frame an open question and think through the options before committing.
Build
Scout, align on a plan, then build under the token and copy rules.
Document
Draft the component's docs — anatomy, states, and API — from the source itself.
Ready
Pass nine quality gates, then publish and lock the component.
Hand off
Bundle the docs, decisions, and prototype so engineering can build.
Every locked call is written to Decisions — append-only Component state tracked on the Readiness dashboard
Lessons and new questions from each step feed back into the next Solve.

From product to engineering

Design sits between product and engineering. Each side hands over a clear artefact, and nothing is duplicated — the handoff is a thin wrapper that points at work that already exists.

Product
Frames the problem
  • Sprint overview and PRD
  • The solve brief and its rationale
  • What success looks like
Design
Shapes the answer
  • Components and page templates
  • Prototypes of the full journey
  • Design language and locked decisions
  • A per-sprint handoff bundle
Engineering
Builds the product
  • Builds from the docs and prototype
  • Raises questions during the sprint
  • Answers get promoted to decisions

The rules underneath

A handful of non-negotiables keep the system coherent as it grows.

Tokens before pixels. Every visual value is a variable — no exceptions.
Solve before build when the question is open; build first when the answer is clear.
Two yeses on the rubric before a pattern earns its own component.
Decisions are append-only — numbered, never edited after closure.
Calm authority. Steady and confident, never urgent or loud.
Plain, active copy. Say "we", use the person's name, no jargon.
