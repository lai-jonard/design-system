How we design

Design workflow

This page explains how design work happens in this system: the layers everything is built on, the steps work goes through from PRD to handoff, and what engineering receives at the end.

The six layers

Everything in the library sits in one of six layers. Each layer builds on the ones below it, so a change lower down carries through to everything above.

Tokens
Design language & decisions
Primitives
Composed components
Page templates
Prototypes
dev/prototypes/Clickable end-to-end journeys — onboarding, document review, signing — assembled from the real components with realistic content. What a founder or investor actually clicks through, tested before anything is finalised.
src/pages/Layout shells — app shell, dashboard, legal-area page. They define where content sits; the components fill them.
src/components/Product-specific compositions — insight card, unlock panel, document list. Each one has a logged reason for existing.
src/primitives/Generic building blocks — button, input, badge, drawer. They carry no product knowledge, so they work in any context.
design-language/ · decisions/The written rules and the numbered log of settled calls — the recorded taste. Everything above is designed against these.
src/styles/tokens.tsColour, type, spacing, radius, and motion values, defined once. Components reference these — no hardcoded values anywhere.

From PRD to handoff

When a PRD comes in, the work runs through the same steps. Each one produces a concrete output before the next starts.

Kickoff
Product and design meet, go through the requirements, and walk the entire flow the PRD describes.
Solve
For new components or patterns only — write the question down and weigh the options. Skipped when documented components already cover the need.
Prototype
Design the flow end to end as a working prototype — assembled from the layers below — components on tokens, following the design language and the logged decisions.
Document
Write docs for anything new and draft the handoff bundle.
Feedback
Product and engineering review the prototype and the draft handoff.
Refine
Work the feedback into the prototype, along with any tweaks to taste. Each settled tweak is logged as a decision.
Ready & hand off
Once everyone is aligned, the work is marked ready and the bundle goes to engineering.
Every settled call — including tweaks made to taste — is logged as a numbered decision and applied to future work Each component’s state is tracked on the readiness dashboard
New questions raised during prototyping or feedback go back into a solve.

Who hands what to whom

Design sits between product and engineering. Each side hands over a defined set of artefacts — the handoff document itself just points at work that already exists.

Product
Frames the problem
  • Sprint overview and PRD
  • The solve brief and its rationale
  • What success looks like
Design
Shapes the answer
  • Components and page templates
  • Prototypes of the full journey
  • Design language and locked decisions
  • A per-sprint handoff bundle
Engineering
Builds the product
  • Builds from the docs and prototype
  • Raises questions during the sprint
  • Answers get promoted to decisions

The standing rules

A few fixed rules apply to all of the above.

Tokens before pixels. Every visual value comes from a token — the rare exceptions are logged, never silent.
Solve before build when a question is still open; build directly when the answer is clear.
Two yeses on the rubric before a repeated pattern becomes its own component.
Decisions are append-only — numbered, and never edited once closed.
Calm visual tone. Steady and quiet — no urgency, no decoration.
Plain, active copy. We say "we", use the person's name, and avoid jargon.
