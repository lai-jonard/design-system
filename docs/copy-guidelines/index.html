Copy guidelines

Rules for all user-facing text in the design system and products built with it.

Language

  • British English always — authorised, organisation, licence (noun), colour, behaviour, defence
  • Active voice always — "We updated your settings" not "Your settings have been updated"
  • Never all-caps. See capitalisation rules below.

Capitalisation

LawAdvisor uses a two-register system. The key question is: does this copy name a surface or control, or does it explain something to the user?

Title Case — structural labels

Use when the copy names a product surface, control, or legal artefact. Every principal word capitalised.

Element Example
Navigation items Governance, Employment, Fundraising
Product area names Cap Table, Due Diligence Pack, Company Manual
Modal titles (feature/screen names) Data Room Settings, Add Employee
Table column headers Document Name, Last Modified, Status
Tab labels Urgent, Proactive, Resolved
Short action CTAs (buttons) Send Invite, Mark Resolved, Review Unconfirmed Entries

Sentence case — guidance and explanatory copy

Use when the copy explains something to the user. First word capitalised, the rest lowercase.

Element Example
Insight headlines Cap table has unconfirmed entries
Insight modal titles (when the title is the description) Articles of incorporation missing founder signatures
Helper text / hints Upload evidence before sharing this pack
"Why this matters" subheadings Why this matters
Empty state text No documents yet. Upload your first to get started.
Section explanations Some shareholders need supporting documents
Body copy, error messages, success moments We couldn't save your changes. Check your connection and try again.

Edge cases

  • Legal proper nouns are always Title Case regardless of context — "Cap Table", "Shareholders' Agreement", "Articles of Incorporation" are named documents, not style choices.
  • Modal titles that describe content (not a feature name) stay sentence case — insight titles fall here.
  • Badge labels are one word — case is moot. Treat as Title Case for consistency.
  • Loading / progress labels ("Resolving…", "Uploading…") are one word — no case decision needed.

Why this pattern — research context

Major design systems (Material Design, Microsoft Fluent, PatternFly) recommend sentence case everywhere for simplicity and localisation. LawAdvisor diverges intentionally: the product vocabulary includes genuine proper nouns (legal documents, commercial terms) where Title Case is standard professional practice. The split-register rule formalises this rather than leaving it implicit.

Calls to action

  • Verb-first, Title Case: "Send for Signature", "Add Employee", "View Details"
  • No jargon — use plain language the user already knows:
Instead of Use
commence start
verify check
submit send
utilise use
terminate end
prior to before
  • One primary CTA per view — never show two primary buttons simultaneously. Use secondary or ghost for additional actions.

Tone and voice

  • Say "we" in product UI — never refer to the product name in third person
  • Always use the person's name — never "the recipient", "the employee", or "the user"
  • Body copy: 1-2 sentences max — if it needs more, the UI is doing too much in one place

Contextual copy patterns

Insight explanations (modal "Why this matters")

Written by the AI legal engine. The explanation surfaces in the insight modal only — the card shows no explanation text.

Ceiling: 3 sentences, 60 words. Structure: (1) what the problem is, (2) why it creates legal or commercial risk, (3) what it affects or what to do next. One paragraph, no lists.

"A shareholders' agreement defines how decisions are made and what happens if a founder leaves. Without one, each founder's rights in a dispute fall back to company law defaults — which rarely match what was agreed. Most investors require a SHA to be in place before closing a pre-seed or seed round."

Never write more than 3 sentences. Do not explain what a document type is in general terms — focus on the specific risk for this company. Avoid hedging language ("may", "could potentially").

Empty states

Inviting and actionable. Tell the user what to do next, not just that nothing is here.

"No documents yet. Upload your first file to get started."

Never: "No data available."

Success moments

Brief, affirming. The user did the thing — acknowledge it and move on.

"Done." "All sorted." "Document sent."

Never: "Your document has been successfully submitted for processing."

Error states

Honest, helpful, unflustered. Say what happened, then say what the user can do.

"We couldn't save your changes. Check your connection and try again."

Never: "An unexpected error occurred. Error code: 500."

Formatting

  • Dates: 15 Apr 2026 (day month year, no commas, abbreviated month)
  • Numbers: use commas for thousands (1,000) and full stops for decimals (3.5)
  • Lists: use sentence fragments, no full stops unless the item is a complete sentence
