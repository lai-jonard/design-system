Prototype standards

The publish gate for flow and page prototypes. A prototype in dev/prototypes/<flow>/ moves into the published docs site (/founder-flow/*.html or equivalent) only when it passes every gate below.

What this is

Prototypes are the structural source of truth engineers read during handoff. They consume layout and composition from a prototype page, then build the functional backing in their own repo. That only works if a "published" prototype is guaranteed to use only published components, style only through tokens, and respect the same standards we apply to components.

Until a prototype passes, it stays in dev/prototypes/ as a work-in-progress. It may still be shared and demoed — it just doesn't become the reference.

Two surfaces, one rule

Path Purpose Gate
dev/prototypes/<flow>/ WIP flow/page mockups — exploration, review, demos None — anything goes
/founder-flow/*.html (and future flows) Published prototypes — engineering's source of truth Must pass every gate below

Moving between them is a one-line change in .eleventy.cjs: add the file to the FOUNDER_FLOW_PAGES passthrough list and rebuild (npm run docs).

The 8 gates

Mirrors the 8 gates used by /audit and /ready, adapted for page-level artefacts.

Gate 1 — Source

The prototype HTML exists at dev/prototypes/<flow>/<NN>-<name>.html. Filename is numbered for flow order. No orphan files.

Gate 2 — Components

Every <la-*> tag used in the prototype is published. A component is published when it has:

  • A page in docs-src/components/
  • An entry in components.md
  • A 🟢 or 🟡 row in handoffs/readiness.md

Any <la-*> that fails this check blocks the prototype. Publish the component first, then retry.

No hand-rolled chrome. No inline <div> styled to look like a button, badge, or card — if the design needs one, it's a missing component, not a prototype problem.

Gate 3 — Design language

Any novel page-level pattern (new layout shape, new composition) is documented in page-architecture.md before the prototype publishes. If the pattern is a one-off, either generalise it or justify it in the prototype's own header comment.

Gate 4 — Behaviour

Interactions, state transitions, and motion go through motion.css easings and durations. No ease-in. GPU-safe only — animate transform and opacity, never height, width, top, left, background, or color. Keyboard interactions work on native semantic elements.

Gate 5 — Responsive

Layout reflows across breakpoints per responsive-design.md. No horizontal scroll at standard widths. Test at least: 1440, 1024, 768.

Gate 6 — Content resilience

No "Lorem ipsum". Use realistic sample content — long names, multi-line descriptions, missing values, edge cases. The prototype should not visibly break when a name is 40 characters instead of 10.

Gate 7 — Accessibility

  • WCAG 2.1 AA contrast across all text
  • Keyboard navigation works end-to-end, tab order is correct
  • Visible focus states on every interactive element
  • ARIA landmarks: main, nav, header, aside where appropriate
  • Colour-plus-something for meaning — never colour alone

Gate 8 — Standards compliance

Tokens. No hex values in the prototype's own <style> block. All colour comes from --la-color-*. Padding, gap, margin use --la-space-*. Radius uses --la-radius-*. Font size uses --la-font-size-*; weight uses --la-font-weight-* per decision 39 (400 default, 500 only when weight change is the state signal, 600 for headings, never 700).

Raw px values. Permitted only on layout-position properties (width, min-width, max-width, height where the dimension doesn't map to a token — e.g. a --signin-panel-width: 711px layout constant). Not permitted for padding, gap, margin, font-size, border-radius.

Copy. Every visible string respects copy-guidelines.md — British English, two-register capitalisation (Title Case for structural labels and CTAs, sentence case for explanatory copy), verb-first CTAs, 1–2 sentences max in body copy.

Assets. Paths are absolute so the Eleventy passthrough works without rewrites:

  • /bundle.js
  • /assets/vendor/phosphor/regular/style.css
  • /motion.css

Links. No dead links. If the flow crosses into a WIP page, link state should make that obvious (disabled, "coming soon", etc.) rather than silently 404.

Anchor link styling. Prototype pages do not load la-base-root.css, so the global anchor styles do not apply automatically. Any <a> element inside prototype-page CSS must replicate the foundation spec manually:

a {
  color: var(--la-color-text-secondary);
  text-decoration: underline;
  text-decoration-color: var(--la-color-border);
  text-decoration-thickness: 1px;
  text-underline-offset: 2px;
  font-weight: var(--la-font-weight-normal);
  transition: text-decoration-color var(--dur-fast) var(--ease);
}
a:hover { text-decoration-color: var(--la-color-primary); }

Scope the rule narrowly (e.g. .rule-disclosure-body a) rather than a bare a {} selector to avoid clobbering navigation links. Full token spec lives in typography.md — Links / anchors.

Stricter than the component gate

A component can ship with a logged divergence — e.g. la-breadcrumb declares a known below-AA contrast on ancestor items, recorded in components.md. Prototypes can't. If a prototype needs a divergence, the component absorbs it first, and the prototype inherits it cleanly through the component.

This keeps prototypes readable as a single artefact: no footnotes, no exceptions to chase across files.

Publishing mechanics

When a prototype passes all 8 gates:

  1. Add the file to the FOUNDER_FLOW_PAGES list in .eleventy.cjs (or create a new passthrough block for a new flow)
  2. npm run docs to rebuild
  3. npm run docs:serve and load the page at the new URL to confirm
  4. Commit — the prototype is now published

To unpublish: remove from the passthrough list, rebuild. The source file stays in dev/prototypes/.

Checklist

Keep this near your tab group when publishing a prototype:

  • [ ] Gate 1 — file at expected path
  • [ ] Gate 2 — every la-* is published
  • [ ] Gate 3 — novel patterns documented or generalised
  • [ ] Gate 4 — motion via motion.css, GPU-safe
  • [ ] Gate 5 — responsive at 1440 / 1024 / 768
  • [ ] Gate 6 — realistic content, no Lorem ipsum
  • [ ] Gate 7 — a11y: contrast, keyboard, focus, ARIA, colour+something
  • [ ] Gate 8 — no hex, no off-scale px, absolute asset paths, no dead links, copy passes
