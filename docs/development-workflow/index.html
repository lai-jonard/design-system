Development workflow

How to develop and contribute to the design system.

Local development

Run the full dev environment with a single command:

npm run dev

This starts three processes together:

  1. tsc --watch — recompiles TypeScript on save
  2. rollup --watch — rebundles dev/bundle.js when dist/ changes
  3. wds --root-dir=dev --watch — serves dev/ and live-reloads the browser

Save a .ts file and the browser reloads automatically within a couple of seconds.

Build chain

src/*.ts  --[tsc]-->  dist/*.js  --[rollup]-->  dev/bundle.js  --[wds]-->  browser reload
  • Source of truth: Lit components in src/
  • Never edit dist/ or dev/bundle.js directly — these are generated outputs
  • Docs pages load bundle.js as an IIFE (no ES module complexity in HTML)

One-off build

For commits or CI, run both steps:

npm run build && npm run build:bundle

Docs pages

Every component has a docs draft in dev/docs-drafts/la-<name>.html (output of /document) which is promoted to docs-src/components/la-<name>.html (the published Eleventy source) by /ready.

  • Open drafts at http://localhost:8000/docs-drafts/la-<name>.html while iterating
  • Each docs page loads ../bundle.js to register all components

Icons in docs pages

Every docs page <head> must include Phosphor CSS to register the icon font globally:

<link rel="stylesheet" href="../assets/vendor/phosphor/regular/style.css">
<link rel="stylesheet" href="../assets/vendor/phosphor/fill/style.css">
<link rel="stylesheet" href="../assets/vendor/phosphor/bold/style.css">

Inside Lit components, shadow DOM does not inherit page stylesheets. Any component using ph-* icons must import ${iconStyles} from '../styles/icons.js' and include it at the top of render(). Browsers cache the font — no extra network cost per component.

DialKit

dialkit/dialkit.js is a floating panel for live CSS variable tuning. Inject it into any docs page to tweak tokens in the browser without touching source files:

<script src="../../dialkit/dialkit.js"></script>

Override window.DIALKIT_DIALS before the script loads to customise which dials appear per page.

Adding a new component

  1. Create the .ts file in the appropriate folder:
    • src/primitives/ for base components
    • src/components/ for specialised components
    • src/pages/ for page-level components
  2. Export the component from src/index.ts
  3. Run /document <name> to generate a draft at dev/docs-drafts/la-<name>.html
  4. Build and verify:
npm run build && npm run build:bundle
  1. Open the draft at http://localhost:8000/docs-drafts/la-<name>.html to check rendering
  2. If the component is Tier 2 (specialised), log the rationale in design-language/components.md

Decisions & evolution

How the workflow itself has changed over time. Each entry: the rule, why it exists, and how to apply it. Promote to a dedicated file if this section exceeds ~10 entries.

Solves are for composed components and flows, not primitives — 2026-04-24

Primitives (src/primitives/) skip straight to /build → /audit → /ready plus atomic decisions/NN-*.md. Composed components (src/components/) and flow/page work get a preceding solves/yymmdd-N-topic/ workspace when the problem is still mushy.

Why: Primitives are narrowly scoped — the questions are atomic ("what weight, what radius, what reflects"). A three-file solve workspace is overkill for a decision that fits in three lines. Composed components and flows span multiple screens and interactions; that shape needs a workspace.

How to apply: If the problem fits one primitive and you can state the decision in a sentence, go straight to decisions/ once it's locked. If the problem spans multiple components, touches UX flow, or needs iteration through drafts, open a solve first. Distilled conclusions get promoted from the solve to decisions/ as they firm up.

Published vs dev — the docs site is the source of truth — 2026-04-24

A component is published when it has JSDoc in source, a page in docs-src/components/, an entry in design-language/components.md, and a row in handoffs/readiness.md. Pre-publish work lives in dev/docs-drafts/ and doesn't count yet.

Why: dev/ is the workbench — anything goes, iterate freely. docs-src/ is the public surface engineering and product read. Without a clear line between the two, stale demos get treated as spec and drift accumulates. The line is: "does it appear on the docs site when I run npm run docs?" If yes, it's published.

How to apply: Components graduate from dev/docs-drafts/ to docs-src/components/ when they pass the /ready gates. Before that, they're WIP and not for external reference. Unrelated to npm publish — this is the docs-visibility bar.

Prototype publish gate — 2026-04-24

Prototypes (flow/page mockups in dev/prototypes/) move into the published docs site (/founder-flow/*.html and equivalent) only after passing every gate in prototype-standards.md.

Why: Prototypes are engineering's structural source of truth during handoff — layout, composition, spacing, interactions. That only works if a published prototype is guaranteed to use only published components, style only through tokens, and respect the same standards we apply to components. WIP prototypes demoed as reference lead to drift.

How to apply: While iterating, keep prototypes in dev/prototypes/<flow>/. When ready to publish, walk the 8-gate checklist in prototype-standards.md. When it passes, add the file to the passthrough list in .eleventy.cjs and run npm run docs. Anything blocked by an unpublished component waits — publish the component first.

Component docs: stage in dev/docs-drafts/, publish on /ready — 2026-04-24

/document writes draft docs pages to dev/docs-drafts/la-<name>.html as standalone HTML. /ready (Gate 9) verifies the draft exists, promotes it to docs-src/components/la-<name>.html in Eleventy-template form, updates the catalogue chip and href, and flips readiness to 🟢. The draft file is deleted after the promotion lands.

Why: dev/ is the workbench, docs-src/ is the published surface. A draft sitting in docs-src/ would already appear on the served site, which muddies the "published" boundary the rest of the workflow relies on. Staging preserves that boundary — drafts are visible under npm run dev but hidden from the built docs site until /ready runs.

How to apply: After /build, run /document <name> to draft the docs page — it lands in dev/docs-drafts/. Iterate on the draft there (viewable at http://localhost:8000/docs-drafts/la-<name>.html under npm run dev). When ready, run /ready <name> — it runs the 9 gates, promotes the draft to docs-src/, updates the catalogue, flips readiness. Then npm run docs to rebuild the site.
