Loading indicators — standard

The system pattern for "the user has waited and we owe them feedback." Covers every place we communicate "something is happening" — from a 200ms button press to a 30-second AI document analysis to a full-page bootstrap.

This doc is the decision layer above the components. The components themselves (la-stepper, la-processing-widget, the shared spinnerStyles) are the implementation. New loading work should pick a category here first, then build, not the other way round.

Source of locked decisions. The build queue, family naming, motion home, and sub-solve list were settled in solves/260428-1-loading-states. Decisions are baked inline below — read that solve for full rationale.

1. Principles

Five principles, each with its because.

1. Match the indicator to the wait, not to the action. A 60ms button click and a 30s AI analysis are different problems. The indicator must reflect how long the user is waiting and whether we know how long it will take, not what they clicked. Same click, different scope of wait, different indicator.

2. The first 1000ms is psychological, not visual. Industry research (Nielsen, Apple HIG) is consistent on the thresholds:

Wait Perceived as Indicator
<100ms Instant None
100–1000ms Responsive but waiting None unless action is destructive — premature spinners feel slower
1–10s Definite wait Spinner / skeleton / inline progress
>10s Long task Determinate progress + time estimate
>30s Background task Defer to a widget the user can leave (la-processing-widget)

The rule that matters: never show a loading state for under 300ms. It flickers and makes fast operations feel broken. Either no indicator at all, or a minimum visible duration (~400ms) once you commit to showing one.

The design system records this rule; it does not ship the infrastructure. The 300ms / 400ms thresholds are guidance for the engineering team. We don't ship a shared MinDurationController from the design system — runtime flicker policy is engineering's call, not ours. Settled in the loading-states meta-solve, Q2.

3. Skeleton over spinner for content; spinner over skeleton for action. Skeletons (placeholder shapes that match the final layout) work for content fetches — list, table, card area — because the user sees the page take shape. Spinners work for user-initiated actions — Save, Send, Resolve — because the user is asking "did my click land?". A spinner over a content area feels lazy; a skeleton over a button is incoherent.

4. Show progress whenever you can show progress. If the system knows how far through it is (file bytes, step count, percentage), use a determinate indicator. Indeterminate spinners are a fallback, not a default. A 20s upload with a stuck spinner feels broken; a 20s upload with 64% feels honest.

5. The indicator owes the user a recovery path if it can fail. Every loading state must have a designed timeout/failure variant. A spinner with no failure path is a trap. The recovery path lives in the component, not in the loading doc — but if the component lacks one, that is a blocker for /ready.

2. Loading taxonomy

Eight loading surfaces in the product. For each: what it is, when it appears, the right component (or the gap if it's missing).

2.1 Inline action spinner

What: A small spinner inside an interactive element — most often inside a button — communicating "I received your click, I'm working on it." Replaces or sits alongside the label.

When: Save, Send for Signature, Resolve, Approve, Add Employee, anything where the user clicked and is waiting on a server round-trip of 300ms–3s.

Component status: Locked — slot 1 in build queue. la-button will gain a loading boolean prop. Sub-solve button-loading-state (to be spawned) settles the exact API: width preservation strategy, double-submit prevention, relationship to disabled. Reuses the shared spinnerStyles — no new keyframes.

Locked behaviour from meta-solve:

  • loading boolean — when true, shows a 12px spinner (--la-spinner-sm), hides the label or replaces icon, sets aria-busy="true"
  • Button must remain non-interactive while loading to prevent double-submit
  • Width must not jump — reserve label width or swap icon-for-spinner of identical size
  • Minimum visible duration ~400ms guidance applies (engineering owns enforcement)

Token: --la-spinner-sm: 12px. Border colour inherits — primary spinner on primary fill becomes white, secondary spinner stays neutral.

2.2 Inline cell / row spinner

What: A spinner that lives inside a row, cell, or list item — used when one item in a list is mutating (rename, delete, reprocess) but the rest are not.

When: Document re-analysis, row-level retry in la-processing-widget, individual list item action.

Component status: Pattern exists informally. Used in la-processing-widget and la-upload-box via the shared spinnerStyles. No primitive wrapping it; consumers compose <span class="spinner" role="status" aria-label="…"> directly.

Rule: The spinner replaces the row's leading icon, not the label. The row text stays readable so the user can tell which row is processing.

2.3 Skeleton placeholder

What: Light-grey blocks shaped like the final content. The user sees the layout take shape, then content fills in.

When: Initial fetch of a content area — insights list on dashboard load, document list on legal-area page open, table data on tab switch. The "first paint of a populated region."

Component status: Locked name — slot 2 in build queue. Primitive will be la-skeleton. Sub-solve skeleton-primitive (to be spawned) settles variants (text / block / circle minimum), sizing API, multi-line text rendering shape, shimmer-on/off rules, and anti-pattern guards.

Locked behaviour from meta-solve:

  • Shimmer motion uses the shared .t-shimmer utility class in motion.css — components import it; they don't define their own keyframes
  • prefers-reduced-motion flattens animation (handled by motion.css)
  • Token: --la-color-bg-muted for the placeholder fill
  • Skeleton shape must mirror the final content layout — a 3-line card skeleton followed by a 1-line card is jank, not a feature

Anti-pattern: A generic grey rectangle with a spinner on top. Pick one — skeleton OR spinner — never both stacked.

2.4 Top-of-page route progress

What: A 2px primary-colour bar that runs across the top of the viewport during route transitions. Common in Linear, GitHub, YouTube. Indeterminate, fast.

When: Page navigation that involves a data fetch — sidebar item click, breadcrumb up, dashboard → legal-area-page.

Component status: Deferred this cycle. Useful for SPA navigation, but the routing patterns in the product aren't settled enough to design against. Remains a documented gap; revisit in a later cycle.

Reference behaviour (for when it returns):

  • 2px height, --la-color-primary, fixed at top of viewport above app shell
  • Indeterminate animation — runs to ~80% in 1s with ease-out, then waits, then completes to 100% on route resolve and fades
  • One global instance (no stacking)
  • Skipped entirely if route resolves in <300ms

2.5 Full-page bootstrap loader

What: The screen shown after sign-in, before the app shell paints. Held only as long as it takes to fetch the user's organisation, legal areas, and initial dashboard data.

When: Cold app boot only. Never on subsequent navigation.

Component status: Deferred this cycle. Once-per-session moment, lower frequency, lower leverage than the four queued primitives. Engineering surface, not a primitive. Revisit in a later cycle.

Reference behaviour (for when it returns):

  • LawAdvisor wordmark centred, with the --la-spinner-sm spinner sized up to ~20px below
  • Single calm copy line: "Loading your workspace…"
  • Surface: --la-color-bg-page. No card, no shadow.
  • Hard timeout at 8s — if bootstrap hasn't completed, switch to a recovery screen ("We're having trouble loading your workspace. Try again or contact support.")

2.6 Async operation widget — la-processing-widget

What: A floating, persistent widget that tracks one or more long-running async operations (typically document upload + AI analysis). Lists each item with its status and lets the user dismiss, retry, or remove.

When: Document upload, AI legal analysis, anything that takes 5–60s and the user can leave. The widget continues to track the work as the user moves around the product.

Component status: Built. la-processing-widget — handles processing | complete | failed per item, collapsible header, dismiss/retry/remove events.

Known gaps (from feedback-moments solve):

  • No designed timeout state — if analysis takes longer than expected, the spinner spins forever
  • Failure copy/recovery exists per row but no aggregate failure summary

Rule: This is the only loading surface that survives navigation. If the wait is >10s and the user might want to keep working, route to this widget.

2.7 Stepped progress — la-stepper

What: A horizontal multi-segment bar showing the user's position in a known-length linear flow. Each segment is one step.

When: Multi-step wizards — unlock flow today, plus any future onboarding, KYC, sign-up.

Component status: Built. la-stepper — 5 statuses (pending | active | complete | skipped | processing), display-only, no events. See decision 42 / unlock-flow-progress solve for status vocabulary rationale.

Rule: The stepper itself is not a loading indicator at the step level — the step body owns its own loading state (typically a button spinner or a la-processing-widget). The stepper communicates position in a flow, not progress within a step.

2.8 Determinate progress — la-progress

What: A horizontal bar that fills 0 → 100% as a known-quantity task progresses. File bytes uploaded, items processed in a batch, export percentage.

When: File upload (bytes known), bulk action ("Resolving 12 of 47 insights"), export ("Generating PDF — 64%").

Component status: Locked spec — slot 4 in build queue. Build-ready; no interim solve needed. Distinct from la-stepper (segmented/discrete) by name and shape.

Locked spec (build-ready):

  • Tag: la-progress
  • API: value: number (0–100); optional label slot; optional description slot for "12 of 47 items"
  • Visual: 4px tall, full-width, --la-radius-pill. Track: --la-color-border-light. Fill: --la-color-primary.
  • Motion: fill via transform: scaleX(value / 100), 200ms --ease-out per value change. GPU-safe.
  • Accessibility: role="progressbar" + aria-valuenow / aria-valuemin="0" / aria-valuemax="100". aria-label or aria-labelledby required on host.
  • Reduced motion: skip transition; jump to value.
  • Out of scope (separate decision if ever needed): striped/animated fill while running, indeterminate mode.

First consumer: la-upload-box — currently shows only a spinner during upload but knows bytes; can show determinate.

Note on naming: the prior file la-progress-bar.ts was renamed to la-stepper.ts precisely because the segmented behaviour was a stepper, not a progress bar. The name la-progress is now used for this determinate primitive.

2.9 AI thinking / streaming state — la-thinking

What: Indicator shown while an AI response is being generated — typically a pulsing dot, a "thinking…" pill, or token streaming.

When: Any AI surface — insight generation, document Q&A, drafting assistance. Distinct from la-processing-widget because the user is actively waiting in-place for the response, not leaving to come back.

Component status: Locked name — slot 3 in build queue. Primitive will be la-thinking. Sub-solve ai-thinking-state (to be spawned) settles the visual default — design space spans pulsing dot, "Thinking…" pill, shimmer over the response area, token-streaming cursor.

Locked behaviour from meta-solve:

  • Pulse motion uses the shared .t-pulse utility class in motion.css; shimmer (when used) reuses .t-shimmer
  • Cross-cuts every AI surface (chat, insights, document Q&A, suggested actions) — primitive must support a clear opt-out for surfaces that already stream tokens

Why this matters for a legal-AI product: AI latency is a structural property of every page that uses an LLM. A consistent, non-anxious thinking-state pattern is more important here than in a non-AI product. See design-language/ai-ux-guidelines.md for adjacent rules.

3. Decision tree

When you hit a wait and don't know which indicator to use, walk this:

Is the wait <300ms?
  └─ Yes → no indicator
  └─ No → continue

Did the user just click something specific (button, row action)?
  └─ Yes → inline spinner (§2.1 / §2.2)
  └─ No  → continue

Are we populating an empty content area?
  └─ Yes → skeleton (§2.3)
  └─ No  → continue

Is this a route transition?
  └─ Yes → top-of-page progress bar (§2.4 — deferred this cycle; until then,
           render a skeleton on the destination page)
  └─ No  → continue

Do we know the percentage / count?
  └─ Yes → determinate progress (§2.8)
  └─ No  → continue

Will the wait exceed 10 seconds and the user can leave?
  └─ Yes → la-processing-widget (§2.6)
  └─ No  → continue

Is this an AI generation?
  └─ Yes → la-thinking (§2.9 — name locked, pattern in flight via sub-solve)
  └─ No  → reach for inline spinner with timeout fallback

If two paths apply, the more specific one wins (skeleton over spinner for content; determinate over indeterminate when count is known).

4. Copy patterns

Pulled from feedback-moments, extended for loading-specific cases.

Rule: present continuous, ellipsis at end, time estimate when known.

State Copy
Generic loading "Loading…"
Workspace bootstrap "Loading your workspace…"
Content fetch (no copy — skeleton speaks)
Document upload "Uploading your document…"
AI analysis "Analysing your document. This usually takes 10–30 seconds."
Bulk action "Resolving 12 of 47 insights…"
Export "Generating PDF — 64%"
AI thinking (synchronous) "Thinking…"
Timeout "This is taking a little longer than usual. We'll keep going."
Failure "We couldn't [verb] this. Try again." (see feedback-moments §6 for recovery copy)

Don't:

  • "Please wait" — paternalistic, says nothing
  • "Loading data" — vague; what data
  • "Working…" — corporate filler
  • All-caps or exclamation marks — never urgent

Action button patterns

For buttons specifically, three valid patterns. The system has a default; consumers can opt into the others when context warrants.

Pattern A — change label to present continuous

The label converts to its -ing form with an ellipsis. Multi-word labels simplify to the primary verb.

Resting label Loading label
Save Saving…
Send for signature Sending…
Add employee Adding…
Approve and propagate Approving…
Generate report Generating…

Use case: high-stakes actions where the present-continuous reads better — signing flows, compliance resolutions, document submissions. Most communicative; requires width preservation (button stays sized to widest state). Vercel defaults to this pattern.

Pattern B — keep label, add spinner (recommended default)

The button label stays the same; a spinner appears in the leading position.

Resting label Loading label
Save (spinner) Save
Send for Signature (spinner) Send for Signature

Use case: most async CTAs. Widely understood, no per-button copy decision, minimal width churn. Linear, Stripe, GitHub, Material default to this pattern.

Pattern C — spinner only, no text

The button shows only a spinner during loading; the label is hidden.

Use case: form submits inside modals/dialogs (the dialog title carries the verb), icon-only buttons, contexts where surrounding text already names the action. Loses information and degrades accessibility (screen reader has no announcement target during loading); use sparingly. Wise defaults to this pattern.

System default: Pattern B. Reasons:

  • Lowest cognitive cost for designers — no per-button loading copy required
  • Matches dominant industry convention (Linear, Stripe, GitHub, Material)
  • Preserves accessibility: existing label still rendered, plus aria-busy on the loading region
  • Avoids width-preservation gymnastics in the common case
  • Doesn't depart from the verb-first CTA copy rule — the verb is already in the label

Pattern A as opt-in for high-stakes legal actions where present-continuous communicates better. Pattern C as secondary for minimalist contexts — form submits in modals, icon-only buttons, dense flows where the surrounding chrome already names the action. C fits the system's calm-authority voice well; we use it more sparingly than B but not so rarely that it's exotic. The exact API for these opt-ins (likely loading-label="Saving…" for A, loading-mode="icon-only" for C) is settled in the button-loading-state sub-solve when it spawns.

5. Accessibility & motion

Accessibility:

  • Every spinner element gets role="status" and aria-label="…" (or visible label) so screen readers announce the loading state
  • Content regions waiting on a fetch get aria-busy="true" on the wrapping element until the fetch resolves
  • Determinate progress uses role="progressbar" + aria-valuenow / aria-valuemin / aria-valuemax
  • Focus must not be moved when a loading state appears — keep the user where they were

Motion (motion.css rules apply):

  • GPU-safe only — transform and opacity. Never width for skeleton fill or progress fill (use transform: scaleX(…)).
  • Shared utilities live in motion.css:
    • .t-shimmer — skeleton shimmer (also reusable by la-thinking if shimmer-on-response-area is the chosen pattern)
    • .t-pulse — AI thinking pulse
    • Components import the class; they don't define their own keyframes. New keyframes require a decision record.
  • Shared spinner lives in spinnerStyles, 800ms linear; every spinner-bearing primitive imports it
  • Progress fill update: 200ms --ease-out per value change
  • prefers-reduced-motion: spinners freeze (already handled in spinnerStyles), .t-shimmer and .t-pulse collapse to static (handled in motion.css), progress jumps without animation

6. Anti-patterns

Concrete failure modes to avoid. Each is a real pattern observed in adjacent SaaS products.

Spinner-in-the-void. A spinner centred on a blank page with no surrounding content. The user has no anchor for what's loading. Replace with a skeleton that matches the eventual layout, or a top progress bar on the populated previous page.

Double indicator. A skeleton AND a spinner. A la-processing-widget AND an inline button spinner for the same action. Pick one — the more specific wins. Two indicators for the same wait look like a bug.

Mismatched skeleton. A 3-row skeleton followed by a 12-row populated list. The skeleton must approximate the real layout; otherwise the page jumps and feels broken.

Eager indicator. Showing a spinner on every action regardless of duration. A 60ms call should not flash a spinner. Either no indicator or a 300ms+ delay before showing one.

Flicker indicator. Showing a spinner that disappears in 80ms because the request returned faster than expected. Once committed, a spinner needs ~400ms minimum visible duration.

Trapped spinner. A loading state with no timeout and no failure variant. If the request hangs, the user is stuck. Every loading surface must have a designed failure path — see §1 principle 5.

Decorative spinner. A spinner on a static surface that isn't actually loading anything ("just to look active"). Spinners are a literal claim that work is happening; lying breaks trust.

Auto-dismissed bootstrap. A full-page loader that times out silently and shows an empty app. If bootstrap fails, say so — see §2.5 for the 8s recovery rule.

8. Sub-solves and open questions

The five blocking opens this doc originally surfaced were settled in the loading-states meta-solve. What remains is two things: the three sub-solves the meta-solve queued, and three advisory opens that need product/engineering input.

Sub-solves to spawn (in build-queue order)

  1. button-loading-state — settles the exact API on la-button (boolean vs. enum, width preservation, disabled coupling, label-swap behaviour). Seed framing exists from prior conversation; reuse on spawn.
  2. skeleton-primitive — settles la-skeleton variants (text/block/circle), sizing API, multi-line text shape, shimmer-on/off rules, anti-pattern guards.
  3. ai-thinking-state — settles la-thinking visual default (pulsing dot vs. "Thinking…" pill vs. shimmer vs. token stream) plus opt-out for surfaces already streaming tokens.

la-progress does not need a sub-solve — its spec is in §2.8 and is build-ready.

Advisory opens (for handoff conversation, not this doc to settle)

  1. Migration ownership. When la-button gains a loading prop, who updates the existing async-CTA call sites — designer flags, engineering executes, or pair? Process question.
  2. la-processing-widget timeout threshold. From feedback-moments solve: suggested 45s default, unverified against real upload + analysis durations. Stays scoped to that solve; this doc surfaces the link.
  3. Where family-wide rules live long-term. This doc absorbs the locked decisions today; a decisions/NN-loading-family-naming.md record is recommended for the locked names so the canonical record outlives the meta-solve workspace.

9. Where this doc lives in the recipe

This is a design-language/ standard, not a /solve and not a single decision. It's the map future loading work plans against. It should be:

  • Updated when a new gap is closed (a 🔴 row flips 🟢)
  • Updated when a new loading surface is discovered the taxonomy doesn't cover
  • Linked from any /solve or /build that touches a loading state, so the work stays consistent
  • Reviewed at sprint end alongside the readiness dashboard

If a future loading question has multiple valid answers, open a /solve against this doc — don't try to settle it inline.
