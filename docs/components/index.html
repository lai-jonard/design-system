design-language/components.md

Components

Working agreement

Before any specialised component is built, the design intent must be defined and logged here.

The question is always: what problem is this component solving?

The answer gets logged in the Tier 2 table before building starts. This is a one-question check — it doesn't block the build, it just ensures the rationale isn't lost.

Base components (Tier 1) are exempt — they follow established conventions and only need rationale logged when we deliberately diverge.

Component taxonomy

There are two tiers of component in this library. Understanding which tier a component belongs to determines how it should be designed, documented, and used.

Tier 1 — Base components (src/primitives/)

Standard UI controls that map directly to established web patterns. These are atoms — single-responsibility, stateless where possible, and not specific to LawAdvisor's product logic.

Rationale approach: These follow established conventions. Use component.gallery as the reference for intended behaviour. No product-specific rationale needed unless we deliberately diverge from convention.

la-activity-item

A single event row in a chronological activity feed — icon circle, a vertical connector line to the next item, a body slot for the event description (may contain inline <a> links), and a timestamp. Compose multiple items in any container; the connector lines chain them into a timeline. Set last on the final item to suppress its connector. Presentational — no events. Used in the per-investor Activity tab (Sprint 12).

Props: time (timestamp string), last (boolean — suppresses connector on the final item) Slots: default (event text), icon (Phosphor <i> inside the circle) Events: none — presentational only

la-avatar

Initials avatar for people and entities. Derives two-letter initials automatically from name (first letter of the first two words), with an initials override for entities or edge cases. Three fixed sizes — sm (32px), md (40px), lg (48px) — map to product contexts (compact lists / person rows / detail headers). Purely presentational and decorative (aria-hidden); the adjacent visible name carries the accessible identity. Used across investor surfaces (Sprint 12) and the share modal.

Props: name (auto-derives initials), initials (override), size ('sm' | 'md' | 'lg', default md) Slots: none · Events: none

la-badge

A small inline primitive with two types. Badge (pill) communicates status, category, or severity at a glance; always paired with content, never standalone. Counter (circle) communicates a numerical count — unread items, queue depth, notification totals.

Types: badge (default), counter Variants: danger, warning, success, neutral, info, accent, primary Appearance (counter only): filled (default), ghost

Badge chrome is shared across variants — all non-primary badge variants use --la-color-bg-subtle surface with a 1px --la-color-border-light border. Only the text colour changes per variant, so the variant distinction reads through typography rather than loud background fills. primary is the single exception — it overrides both bg and border to the solid primary colour (used sparingly for prominent labels). See decision log for rationale.

Counter chrome is variant-filled — counters use solid variant colours (filled) or tinted --la-color-surface-* fills (ghost). No border. This keeps counters visually louder than badges, appropriate for numerical signals.

Variant When to use
danger Urgent issues requiring immediate action
warning Proactive issues — not urgent but should be addressed
success Resolved items, positive states
neutral Locked, inactive, or unknown state
info Processing, in-progress states
accent Category labels (e.g. "Doc Quality", "Legal Compliance") — badge type only, not a status
primary Prominent count or label against brand-primary context — default counter colour
Type When to use
badge Status or category label. Text-only or with a leading Phosphor fill icon (ph-fill ph-*).
counter Numerical count signal next to a title, nav item, or inbox. Single or multi-digit.

With tooltip: when a status concept may be unfamiliar to users (e.g. "DD window", "Processing"), wrap the badge in la-tooltip with appearance="light" to give context in one sentence. Dark tooltips feel too heavy against small badges.

la-breadcrumb

A depth signal rendered in the page header. Shows the user's location in the navigation hierarchy and provides a one-click path back up the tree. Used inside la-app-shell via the breadcrumbs prop — not rendered standalone.

Props: items (BreadcrumbItem[]) — ordered array from root to current page. Each item is { label: string; href?: string }. Last item is the current page (no link).

Collapse behaviour: At 4 or more items, the trail collapses to root / ••• / parent / current. Tapping ••• (an la-icon-button) expands the full trail. Collapsing resets automatically when items changes.

Colour: Ancestor links use --la-color-text-faint; the current page label uses --la-color-text-muted. Both are below WCAG AA at small text sizes — accepted divergence for navigational support copy. Hover on ancestor links resolves to --la-color-text (passes AA).

Docs-only — see docs-src/components/la-breadcrumb.html.

la-button

The primary interactive control. Every call-to-action uses this. Always verb-first label, sentence case.

Variants: primary, secondary, success, danger, ghost Sizes: sm (28px), default (36px), lg (44px)

Variant When to use
primary The main action on a screen or panel. One per view.
secondary Cancel, back, or secondary actions alongside a primary
success Approve, complete, or confirm positive outcomes
danger Destructive actions — delete, remove, reject
ghost Low-prominence links or tertiary actions
Size When to use
sm Dense layouts — table row actions, widget headers, toolbar buttons
default The standard for almost every button across the product
lg Hero CTAs and large forms — unlock flows, onboarding, primary empty-state actions

Labels are consumer responsibility — la-button does not truncate. Keep labels short and verb-first. If a label needs to be long, the call-to-action isn't clear enough.

Loading state: the loading boolean prop turns any button into an async-CTA-ready button. Pattern B is the system default (label stays, spinner shown alongside) — opt into Pattern A with loading-label="Saving…" for high-stakes actions, or Pattern C with hide-label for minimalist contexts. Full guidance in design-language/loading.md §4.

Loading prop Type Default Notes
loading boolean false Suppresses click, sets aria-busy="true". Orthogonal to disabled.
loading-label string '' Pattern A — replaces default content during loading (e.g. "Saving…").
hide-label boolean false Pattern C — hides default content during loading; only spinner visible. Wins over loading-label if both are set.

Auto-route by icon-left: the spinner placement adapts to the button's content. Buttons with an icon-left slot show the spinner in place of the icon during loading (label stays). Buttons without an icon-left show the spinner centred in the label area (label fades). In both cases the button keeps its resting width — no phantom space, no jump on toggle. Pattern C with hide-label collapses the leading area entirely and centres the spinner regardless of icon presence.

Pairing with la-spinner: la-button embeds la-spinner internally. Spinner size and appearance are auto-mapped — sm-button → sm-spinner; default + lg → md-spinner; primary/success/danger variants → appearance="invert"; secondary/ghost → appearance="default". Consumers never set spinner props directly.

la-icon-button

An icon-only button for secondary actions in dense surfaces — widget headers, toolbars, sidebar toggles. Shares the --la-button-height-* size scale with la-button so mixed rows align at any size. Requires a label prop for screen readers (no visible text).

Variants: default, primary, outline, danger Sizes: sm (28px), default (36px), lg (44px)

Variant When to use
default Most common — transparent with hover fill. Widget headers, toolbars, inline controls
primary High-emphasis single action, used sparingly
outline Secondary emphasis — between default and primary
danger Destructive actions — delete, remove
Size When to use
sm Widget headers, dense toolbars, actions inline with text rows
default Standalone icon buttons in standard-density layouts
lg Touch targets or hero icon actions — rare

Use la-button with its icon-left or icon-right slot when you need an icon and a text label. la-icon-button is specifically for icon-only triggers.

la-input

A text input for forms and modal dialogs. Wraps a native <input> element with a consistent label, hint, icon slot, and state system. Single size — layout is determined by the container.

Types: text, email, password, search, tel, url, number

Prop When to use
label Always set for named form fields. Omit only for search inputs where the icon and placeholder provide sufficient context.
hint Use for format guidance ("DD/MM/YYYY") or constraints ("As registered with ASIC"). Turns red in error state.
required Appends * to the label and forwards required to the inner input. Use on mandatory fields.
error Set programmatically after validation. Pair with a hint that explains what's wrong.
disabled Pre-filled read-only fields — e.g. jurisdiction locked to a value.

Icon slots: icon-left for context (envelope, person, lock), icon-right for actions (password reveal) or status (validation check). Padding adjusts automatically — no extra props needed.

Search inputs are a separate component (la-search-input, not yet built). Use type="search" on la-input as a temporary measure only — the native clear button is not styled.

la-menu + la-menu-item + la-menu-toggle + la-menu-divider

Tertiary action surface — a 3-dot trigger button that reveals a floating list of actions. la-menu owns the trigger, panel, and open/close lifecycle. la-menu-item is the action row primitive: label, optional icon, and a danger variant. la-menu-toggle is a switch row for boolean in-menu settings (does not close the panel on change). la-menu-divider is a visual separator between sections. Status: 🟢 v1 ready (2026-05-07) — full reference at docs-src/components/la-menu.html.

Problem solved: Every page or card surface needs a consistent home for tertiary actions (rename, delete, export, etc.). Before this, teams were inventing ad-hoc popouts with inconsistent styling, shadow, and keyboard behaviour. la-menu is the one canonical answer — reach for it whenever an action doesn't warrant a visible button and isn't the primary CTA.

origin property: Controls which corner the panel scales from. Set to match the trigger's position on the page — top-right (default) for a 3-dot button in the top-right of a card, bottom-right if the trigger is near the viewport bottom.

Events: la-menu-open / la-menu-close (from la-menu), la-menu-select with {label, value} (from la-menu-item, bubbles), la-menu-toggle-change with {label, value, checked} (from la-menu-toggle, bubbles — does not trigger panel close).

la-select

Custom single-select combobox. Trigger chrome mirrors la-input (height, border, focus ring, error, label, hint) so a select sits naturally in a form row alongside inputs. Slotted <la-option> children carry the values; rich content (icons, multi-line) is allowed inside each option. Status: 🟢 v1 ready (2026-05-05) — full reference at docs-src/components/la-select.html.

Why custom (not native <select>): native renders the open dropdown via the OS, which means we can't style options, hover, selected state, or icons inside the panel — every product surface would render differently across macOS / Windows / Linux. Custom is more code but gives us full token control and a consistent feel.

Prop When to use
label Always set for form fields.
placeholder Shown in the trigger when no value is selected. Defaults to 'Select…'.
hint Format / constraint guidance, same convention as la-input.
required / error / disabled Mirror la-input.
value Controlled — consumer updates from la-change.
name Form field name. Reserved for future ElementInternals form-association (out of scope in v1).

Partner — la-option: slotted child carrying value (required), disabled, selected (reflected by parent — do not set directly). Default slot accepts plain text or rich content (icons, multi-line); plain-text fallback used for type-ahead matching via textContent. No own events — selection always travels through the parent's la-change.

Selection glyph: the inline check on the chosen option is ph-check (plain). Distinct from ph-check-circle which is reserved for the success status — see iconography.md § Form selection and decision 59.

Keyboard: ↑/↓/Home/End cycle, Enter/Space select, Esc closes, Tab closes-and-progresses, type a printable character to jump to the next option starting with that letter. Type-ahead matches single characters only — no buffer in v1.

Positioning: panel floats below by default. When there's not enough space below (e.g. select sits near the viewport bottom), it flips above. Panel uses position: absolute inside the host's shadow DOM — an ancestor with overflow: hidden will clip it. v2 candidate: switch to the Popover API for top-layer rendering when an overflow-clip case appears.

Out of scope (deferred to v2 with a solve): multi-select, search input inside the panel, ElementInternals form-association.

la-select vs filter-menu — when to reach for which

Use la-select when Use a filter-menu (overlay of la-checkbox) when
Picking exactly one value Picking multiple values
The select sits in a form row alongside inputs The trigger sits in a toolbar next to other action buttons
Default state shows the chosen value as text in the trigger Default state shows a count ("3 selected") or a pill list
The selected value is the row's data ("Jurisdiction: Australia") The selected values filter or filter-out other content

If the answer is "single + form-row," reach for la-select. If it's "multiple + toolbar/filter," compose la-checkbox rows inside an overlay (no current dropdown primitive for this — pattern is documented in la-data-table's filter-menu snapshot).

la-stepper

A stepped progress indicator for multi-step flows. Shows where the user is in a sequence and what's been completed, skipped, or failed. Used in the unlock flow.

Step statuses: pending, active, complete, processing, skipped, failed

Each step is a numbered circle. Use when a process has a defined sequence with discrete steps the user must move through in order.

la-spinner

Indeterminate loading indicator — a small spinning ring used inside buttons, rows, and async-operation surfaces to communicate that work is in progress. Distinct from la-stepper (linear flows with discrete steps) and la-progress (determinate, known-quantity tasks). Slot 1 of the loading-states family — see design-language/loading.md for the family roadmap.

Props: size ('sm' | 'md' | 'lg', default 'sm'), appearance ('default' | 'invert', default 'default'), label (string, default 'Loading').

Sizes: sm 12px · md 16px · lg 24px. Pairing with la-button: sm-button → sm spinner; default-button and lg-button → md spinner.

Appearance: default inherits colour via currentColor from any ancestor. invert overrides to --la-color-text-invert (white) for dark backgrounds — use on filled primary / danger button variants where the foreground is white.

Use indeterminate (this primitive) when the system can't tell how much work remains; reach for la-progress when bytes / count / percentage are known.

la-card

A surface container used to group related content. Provides a visual boundary (border + background + radius) without adding layout logic. Takes an optional heading prop for a titled header section.

Props: heading (string), nopadding (boolean)

Use nopadding when the content inside the card manages its own spacing (e.g. a list that needs edge-to-edge rows).

la-list-item

A single row within a list. Handles layout for icon + content + meta pattern. Draws a bottom border between items automatically; last child has no border.

Props: error (boolean — highlights row in red tint)

Use inside la-card or any list container when rows have a consistent icon + label + metadata structure.

la-choice

A selectable option row with a label slot and optional description. Clicking toggles checked internally and fires a la-change event. Used in onboarding and unlock flows for single boolean choices, single-select question steps (choice), and multi-select question steps (multi-choice). Composes la-checkbox internally — the inner checkbox is purely visual (pointer-events: none, aria-hidden="true"); the host element is the sole accessible control.

Props: checked (boolean) · invalid (boolean) · description (string) · variant ('default' | 'contained')

Events: la-change — { checked: boolean } — fires on click or Space/Enter keypress, bubbles and is composed.

Variants:

  • default — plain row, no background or border. Parent controls spacing via gap.
  • contained — subtle background (--la-color-bg-subtle), --la-radius-xl border radius, --la-space-md padding. Adds a --la-color-primary border when checked. Hover overlay via ::before (GPU-safe opacity transition).

States: invalid applies a danger ring to the checkbox wrapper and danger text colour to the label. Independent from checked — a row can be both, either, or neither.

Groups: For single-select (choice) behaviour, wrap in role="radiogroup" and enforce exclusivity in the parent la-change handler. For multi-select (multi-choice) behaviour, wrap in role="group" and let each tile toggle independently. Both patterns are field types in la-unlock-panel.

Docs: docs-src/components/la-choice.html. Renamed from la-checklist — see decision 64.

la-checkbox

A bare form checkbox — native <input type="checkbox"> wrapped in a tiny Lit element, tokenised, with an indeterminate state. No label, no description, no row chrome. Used where the surrounding UI owns the labelling.

Props: checked (boolean) · indeterminate (boolean) · disabled (boolean) · label (string — aria-label)

Events: la-change — { checked: boolean } — fires on user toggle only, not programmatic updates.

Canonical uses:

  • Table row selection (composed inside la-data-table for multi-select)
  • Filter menus / "show columns" popovers / sort dropdowns — dense menu items
  • Bulk-action toolbars (the header "select all" trigger)
  • Dense form grids where labels live in external grid cells

la-switch

A binary on/off toggle. Wraps a native <input type="checkbox" role="switch"> styled as a pill track + circular thumb. Form submission and screen-reader semantics (role="switch", aria-checked) come from the platform; visuals come from tokens.

Props: checked (boolean) · disabled (boolean) · label (string — aria-label for the input)

Events: la-change — { checked: boolean } — fires on user toggle only, not programmatic updates.

States: off · on · hover (off: darkens to --la-switch-bg-off-hover + --la-shadow-sm; on: darkens to --la-color-primary-hover) · focus-visible (ring on the track via --la-focus-ring-*) · disabled (opacity: 0.4 + cursor: not-allowed).

Canonical uses:

  • Settings / preferences that apply instantly on click (email notifications, public profile, sticky sidebar)
  • Feature flags in admin panels
  • Filter toggles in toolbars where the change takes effect immediately

Divergences / known constraints:

  • Sizing hardcoded (32×18 track, 14px thumb). Single size in v1. If sm/lg variants ship, graduate to a --la-switch-height-* token family following the --la-info-hint-* precedent.
  • Dedicated track-bg tokens. Off-state track uses --la-switch-bg-off (#E2E3E4) and --la-switch-bg-off-hover (#D1D5DB) — the shared --la-color-bg-* scale tops out at a warm light grey that doesn't give the switch enough presence against white surfaces. On-state reuses --la-color-primary / -primary-hover from the shared scale.

la-switch vs la-checkbox — when to use which

Use la-switch when Use la-checkbox when
The change applies instantly — settings, feature flags, preferences The value is staged — part of a form submitted later
The control is binary and permanent in UI A multi-select or list selection
You want SR users to hear "on"/"off" You want SR users to hear "checked"/"unchecked"
A clear "off" state is a meaningful UI affordance An indeterminate state is possible

la-checkbox vs la-choice — when to use which

The rule: la-choice is a statement. la-checkbox is a control.

Context Reach for Why
Onboarding / unlock steps with descriptions la-choice The row is the moment of attention — built-in label + description, cushioned padding, contained variant
Consent & agreement lines ("I agree to the terms") la-choice Full-row attention matches the weight of the decision
Settings-style forms (one label per row, maybe a description) la-choice Each row tells a story
Table row selection la-checkbox The table owns the row — checkbox is one control among many cells
Filter menus, sort dropdowns, "show columns" popovers la-checkbox Menu rows are dense (28–32px) inside an overlay — la-choice's cushioned padding would feel bloated
Bulk-action toolbars la-checkbox Inline control next to actions; no description needed
Form grid cells ("Admin?" column in a team table) la-checkbox Grid structure owns the labelling

Edge case to watch: a filter-menu row "looks like" a choice item (checkbox + text). The context decides — menu rows are persistent-overlay-dense, choice rows are cushioned-persistent-inline. Using la-choice in a menu would make the menu feel heavy.

la-panel

A generic content container with a subtle background, used for secondary information blocks within a layout. Less prominent than la-card — sits inside a page rather than being the primary content surface.

Props: heading (string — optional label rendered above slotted content), clickable (boolean — adds pointer cursor), dimmed (boolean — reduces opacity to 0.7)

Use dimmed for locked or unavailable states. Use clickable when the panel should indicate interactivity — wire a native click listener on the host; wrap in a <button> or <a> if keyboard semantics are required.

la-tooltip

A hover- or focus-revealed tip that anchors to any element. Use for short, supplementary context — the meaning of an icon-only button, format guidance on a field, or a label's expansion.

Props: tip (string — the tip text), position ('top' | 'bottom' | 'left' | 'right' — which side of the trigger the tip appears on), appearance ('dark' | 'light' — inverted surface by default, white surface with border + subtle shadow for light), maxWidth (string — any CSS length; when set, switches the tip from nowrap to wrapping at width: max-content capped by the value)

Appearance pairing: Use dark for transient action labels on icon buttons (the louder default). Use light for quieter inline contextual help — the la-info-hint component wraps this appearance by default.

Wrapping: Default is white-space: nowrap — a single line. Long tips clip against the viewport edge unless you set max-width. For anything longer than a short phrase, pass max-width="240px" (or any CSS length) so the tip wraps cleanly.

Keyboard reveal: The tip is revealed on :host(:hover) and :host(:focus-within), so keyboard users see it when they Tab into a focusable trigger. Slot a focusable element (<button>, <a>, or anything with tabindex) — non-interactive text won't trigger focus-within.

v1 a11y gaps (deferred): No aria-describedby wiring between trigger and tip, so screen-reader users do not hear the content. No prefers-reduced-motion override. Do not place critical information inside a tooltip until these close.

la-alert

Inline alert bar for state-driven, contextual messages. Persistent — not dismissible. Stays visible until the state it describes changes. For transient feedback that auto-dismisses, reach for la-toast. (Welcome / onboarding hero surfaces previously pointed at la-welcome-banner; that component was descoped from Sprint 2 on 2026-05-05 — no canonical home in any current sprint solve. Source remains in the tree if a future sprint adopts the welcome-first proposal.)

Props: variant ('info' | 'success' | 'warning' | 'danger' | 'neutral' — sets the surface + text colour pair)

Slots: icon (optional leading glyph — always use Phosphor fill weight ph-fill ph-* to match the solid tinted surface), default (message body — wraps naturally)

Variant When to use
info Processing, in-progress, awaiting states
success Resolved, confirmed, up-to-date states
warning Proactive heads-up — not urgent but should be addressed
danger Urgent issue requiring immediate action
neutral Read-only context, no action available

Responsive behaviour: below 240px container width, the alert stacks (icon above body) via a container query on :host. No props, no viewport checks — responds to the parent's available width.

Icon-text optical alignment: the icon's line-box is matched to the body's single-line line-box so the two characters share the same optical centre — crisp single-line alignment without breaking multi-line top-alignment.

v1 a11y gap (deferred): no role="alert" or role="status" on the host, so screen readers will not announce the alert when it appears. Do not use for SR-critical messages until this is added.

la-toast

Transient notification for outcome feedback. Auto-dismisses after a set duration (default 3s) or on close-button click. Use for past-tense moments — "Document uploaded.", "Changes saved.", "Contract sent for signature." For persistent state that needs user attention, reach for la-alert. For stacked multi-toast management (positioning, staggered motion, swipe-to-dismiss), place la-toast-stack once in the layout and trigger via the toast() API — la-toast is the primitive underneath it.

Props: variant ('default' | 'success' | 'warning' | 'danger'), title (primary text), message (optional second line), open (visibility, reflected), duration (auto-dismiss in ms; 0 disables when la-toast-stack owns the timer)

Slots: icon (leading glyph — Phosphor ph-fill weight, variant-coloured), default (fallback for message when inline markup is required)

Events: dismiss (bubbles) — fires on close-button click or auto-dismiss. Host flips open = false before dispatching.

Methods: show() — flips open = true and starts the auto-dismiss timer when duration > 0.

Variant When to use
default Neutral outcome — "Saved.", "Done."
success Positive outcome — "Uploaded.", "Sent."
warning Soft heads-up — "Unsaved changes.", "Connection slow."
danger Negative outcome — "Upload failed.", "Couldn't save."

Copy rule: British English, past tense, 1–2 sentences max, sentence case, ending in a full stop.

Motion: owned by la-toast-stack (via Web Animations API — entry, stacking, and exit are imperative on the stack, not the toast). la-toast has no intrinsic enter/exit animation by design — standalone usage snaps in/out and is not the intended consumption path. Reduced-motion is handled in the stack's branch.

a11y: role="status" + aria-live="polite" auto-set on connect for default/success/warning so screen readers announce without interrupting. variant="danger" upgrades to role="alert" + aria-live="assertive". Close button uses la-icon-button with label="Dismiss".

la-modal

Dialog overlay for confirmations, forms, multi-step flows, and document review. Blocks the rest of the page with a scrim, takes visual focus via scale + fade animation. Persistent until the user dismisses — closes on the X button or backdrop click.

Props: open (boolean — controls visibility, reflected), heading (string — the title), description (string — optional subtext below the heading, muted), size ('default' | 'lg' — 560px / 640px max-width)

Slots: default (body content — scrolls internally when content exceeds panel height), footer (action row — auto-hidden via :empty when nothing is slotted)

Events: close (bubbles, not composed) — dispatched on X click and backdrop click. Also flips open = false internally.

Size When to use
default Confirmations, short forms, most cases (560px max)
lg Document review, longer intake forms, wider content (640px max)

Chrome: radius --la-radius-xl2 (24px), shadow --la-shadow-lg, scrim --la-color-scrim (20% black). Panel is a flex column — header and footer are pinned, body scrolls internally at 90vh max.

Padding rhythm:

  • Content regions (heading, body, footer buttons) all at --la-space-xl (24px) from the panel edge
  • Close button sits in a tight --la-space-lg (16px) top-right pocket
  • Heading-block has a 4px top nudge (--la-space-xs) for optical alignment with the close button

Motion: open + close both animate. Backdrop fades in/out; panel scales between --scale-enter (0.96) and 1. Duration --dur-modal (250ms), curve --ease-out. GPU-safe (transform + opacity). Collapses to 0ms under prefers-reduced-motion: reduce.

Footer separator is conditional — the rule above the footer only appears when body content is scrollable (a [scrollable] attribute auto-toggled via a ResizeObserver on the body). Short content = no separator; long content = separator signals "more above".

Close button uses la-icon-button size="sm" with ph-x — follows the iconography canonical close mapping and a11y label rule.

a11y — closed in v1: host sets role="dialog" + aria-modal="true" on connect; Escape key closes; close button uses la-icon-button with accessible label.

a11y — still deferred: no focus trap inside the panel, no return-focus-to-trigger on close. Tab can move focus into the page behind the modal. Fix before using for keyboard-primary flows. la-drawer already ships both — use it as the reference implementation when upgrading la-modal.

la-drawer

Edge-anchored dismissible overlay panel. Slides in from any edge (right, left, top, bottom). Used for detail inspectors (right), mobile navigation (left), and any secondary surface that appears over the page without taking focus permanently. For centre-anchored confirmations or forms, reach for la-modal. For stackable transient feedback, reach for la-toast / la-toast-stack.

Props: open (boolean — controls visibility, reflected), edge ('right' | 'left' | 'top' | 'bottom' — default 'right', reflected), size (string — CSS size like "320px" or "50vw"; falls back to --la-drawer-size when unset), heading (string — optional title), description (string — optional muted subtext), label (string — aria-label when no heading), backdropDismiss (backdrop-dismiss — opt-in backdrop-click close, off by default), noScrollLock (no-scroll-lock — opt-out of page scroll lock, on by default)

Slots: default (body content — scrolls internally), header (replaces heading block; close button always stays), footer (action row — auto-hidden via :empty)

Events: close (bubbles, not composed) — fires on Escape, close-button click, or backdrop click when backdrop-dismiss is set. Host flips open = false before dispatching.

Edge When to use
right Detail inspectors — click a row, inspector slides in
left Mobile navigation — sidebar content moved into a drawer on narrow viewports
top Notification-style surface
bottom Mobile-sheet surface

Chrome: radius --la-radius-xl on the inward-facing corners only, shadow --la-shadow-lg, scrim --la-color-scrim (shared with la-modal). Panel is a flex column — header and footer are pinned, body scrolls internally.

Sizing: --la-drawer-size defaults to --la-aside-width (320px). Width (right/left) or height (top/bottom) caps at 85vw / 85vh on regular viewports. Below 480px viewport, right/left drawers become 100vw and their corner radius collapses to zero — top/bottom are already 100vw at every width so the rule doesn't apply to them.

Motion: open + close animate via CSS transitions on transform + opacity (GPU-safe). Panel slides on transform with --ease-drawer × --dur-modal (250ms). Backdrop fades with --ease × --dur-modal. Visibility flip at transition end kills the lingering shadow when closed. Collapses to 0ms under prefers-reduced-motion: reduce.

Dismissal: Escape key and close button always dismiss; backdrop-click dismissal is opt-in via backdrop-dismiss (off by default — protects users from losing progress during multi-step tasks).

Focus management: on open, focus moves to the first focusable element inside the panel (the close button by default). Tab/Shift+Tab cycles only within the panel (trap). On close, focus returns to the element that opened the drawer. This closes the same a11y gap la-modal still has — la-drawer is prior art for a future la-modal upgrade.

Scroll lock: page behind the drawer is overflow: hidden while open by default; no-scroll-lock opts out for reference-style drawers where users cross-check content.

Composition in la-app-shell: on narrow content areas (container ≤ 1023px), la-app-shell internally swaps its inline aside panel for la-drawer edge="right". Consumers slot into slot="aside" and dispatch la-toggle-aside to open — no direct la-drawer instantiation needed for that case.

Close button uses la-icon-button size="sm" with ph-x — follows the iconography canonical close mapping.

a11y — closed in v1: host sets role="dialog" + aria-modal="true" on connect; Escape closes; close button uses la-icon-button with accessible label; focus trap + restoration both implemented (unlike la-modal).

la-table

A presentational scroll-wrap around a consumer-provided <table>. The component owns horizontal overflow and a single <slot>; everything inside (the <table>, <thead>, <tr>, <th>, <td>) is authored in light DOM by the consumer. Default row/cell styling is supplied by the shared dev/la-table.css stylesheet.

Props: none. Slots: default (must contain a real <table> element). Events: none.

Required consumer markup — real <table> inside:

<la-table>
  <table>
    <thead>
      <tr><th>Document</th><th>Status</th></tr>
    </thead>
    <tbody>
      <tr><td>…</td><td>…</td></tr>
      <tr class="success"><td>…</td><td>…</td></tr>
    </tbody>
  </table>
</la-table>

Why the wrapper isn't smarter: HTML parsing strips stray <thead> / <tbody> / <tr> / <th> / <td> tokens when they aren't inside a real <table> — by the time shadow DOM projection runs, those elements are gone. A real <table> preserves them. The component accepts this constraint rather than inventing a custom DOM.

Dependencies — load on any page that uses la-table:

  • dev/la-tokens-root.css — the :root token declarations
  • dev/la-table.css — default chrome for la-table table, thead th, tbody td, :hover, tr.success, last-row border suppression. All tokenised.

Row state via class: tr.success adds a tinted --la-color-surface-success background for positive row states (e.g. a document that just finished processing). No other row classes in v1 — consumers add their own.

la-data-table

A data-driven table primitive. Consumer passes columns and rows as properties; the component owns sort, selection, row-click dispatch, sticky header, density, and the empty / loading states. Everything is controlled — sort and selected arrays flow in via properties and flow out via events, so the consumer owns the state.

Props:

  • columns (DataTableColumn<T>[]) — column definitions. Each column: id, label, optional align, width, sortable, render(row).
  • rows (T[]) — the dataset.
  • rowKey (keyof T | (row: T) => string, default 'id') — stable row identifier used by selection and sort.
  • sort ({columnId, direction: 'asc' | 'desc'} | null) — current sort. Controlled — consumer updates from la-sort-change.
  • selected (string[]) — currently-selected row keys. Controlled — consumer updates from la-selection-change.
  • selectable — 'none' (default) / 'single' / 'multi'. multi adds a checkbox column + header select-all.
  • density — 'compact' / 'default' / 'comfortable'. Controls vertical and horizontal row padding via --la-dt-row-pad-v / --la-dt-row-pad-h.
  • sticky-header (boolean, default true) — pins the header when the wrap scrolls.
  • loading (boolean) — replaces the body with skeleton rows (or the loading slot).
  • row-clickable (boolean) — pointer cursor on every row without enabling selection; use when rows are navigable.

Slots: empty (overrides default "No results"), loading (overrides default skeleton rows).

Events:

  • la-sort-change — {columnId, direction} | null. Fires when a sortable column header is activated. Cycles asc → desc → off.
  • la-row-click — {row, rowKey}. Fires on any row click; in selectable='single' this also toggles selection.
  • la-selection-change — {selected: string[]}. Fires on checkbox toggle, header select-all, or single-row click.

States: default row, hover row, selected row, sortable header (idle / hover / sorted asc / sorted desc), focus-visible on sortable header (Enter/Space activates), loading (skeleton shimmer, opacity-only), empty ("No results" or consumer slot).

Composition: uses la-checkbox inside shadow DOM for the header select-all and each row's checkbox in multi mode. Row-checkbox clicks stop propagation so they don't trigger la-row-click.

Accessibility: native <table> / <thead> / <tbody>. Sortable <th> gets role="button", tabindex="0", and aria-sort="ascending | descending | none". Keyboard Enter/Space activates sort. Non-sortable headers stay role="columnheader". Sort-direction icon is aria-hidden="true". Composed checkboxes get label props ("Select all rows", "Select row {key}").

Motion: row hover + sortable header colour transitions use --dur-fast × --ease. Skeleton shimmer is an opacity keyframe (GPU-safe). @media (prefers-reduced-motion: reduce) kills transitions and the shimmer.

Gate 8 noted exceptions: width: 40px on the checkbox cell + <col> and height: 12px on skeleton rows are hardcoded — no current token for checkbox-column width or skeleton-row height. Matches the la-checkbox / la-upload-box precedent for glyph-detail pixel values.

Known v1 limits: no virtualisation (every row renders), no column resize / reorder, no inline filter UI (sort only), no row expansion / subrows, no selection-preserving re-sort beyond what the consumer maintains via selected + rowKey. These are explicit deferrals — the primitive is for "dataset rendering," not "full spreadsheet."

la-table vs la-data-table — when to reach for which

Rule of thumb: write markup if the rows are static or under 10; pass data if the rows are a dataset.

Use case Component
Pricing or spec comparison with hand-written rows la-table
Side-by-side feature matrix where cells carry bespoke content la-table
Settings page with fixed, labelled rows la-table
Document list, audit log, invoices, team roster — any backend-sourced dataset la-data-table
Anywhere you need sort, multi-select, sticky header, row-click events, empty / loading states la-data-table

la-table is a positioning wrapper; la-data-table is a data-driven component. They solve different problems and neither subsumes the other.

la-tabs

Single-selection tab list. One tab is active at a time; its associated panel renders, others are hidden. Three components compose the primitive — la-tabs owns active state and keyboard, la-tab is a trigger, la-tab-panel is a content region.

Implemented via a ::before pseudo-element rendering the visible chip; the host carries the larger interactive area. la-tabs[vertical] vs la-sidebar — when to use which

Both are vertical lists with single selection. They serve different scopes:

Component Scope Variant focus
la-tabs[variant='vertical'] In-page tab pattern. Panels swap in place; no URL routing. Lives inside a card, modal, settings dialog. Carries role="tablist" semantics. Sizes, density (in-page concerns)
la-sidebar App-shell navigation. Items route to different views/URLs. Lives at the edge of the app. Collapsed icon-only, brand area, footer slot, multi-section grouping (app-shell concerns)

This split matches Material UI, Radix, Ant Design, and Chakra — every major design system supports vertical orientation on its tabs component while keeping the app sidebar separate. The variant sets are intentionally different — sidebar's collapsed-icon mode is meaningless inside a settings panel, tab sizes are meaningless on a full-height app rail. See decision 43.

Hierarchy rule — pick the variant that matches the navigational tier:

  1. segmented (most prominent): page-level / primary content nav
  2. underline: legacy chrome only — reach for it second
  3. pill: section-level inside a card or panel
  4. vertical: orthogonal layout choice, not a prominence step

Don't nest the same variant in itself; if a panel needs tabs, drop down to pill. Don't pair siblings of different tiers in the same band. Don't reach for vertical to mean "smaller" — it's a layout call, not a size call.

v1 a11y baseline — full ARIA tablist semantics (role, aria-selected, aria-controls, aria-labelledby, aria-orientation), roving tabindex, focus rings on all variants, multi-channel active state (colour + indicator). Reduced-motion auto-respected via --dur-base token cascade.

Known v1 limits — no explicit mobile / narrow-width handling: long labels overflow horizontal rails (intentional, documented in "Long labels" snapshot, locked to variant="underline" because the overflow shape is rail-specific). No auto-scroll, no collapse-to-more-menu in v1. Disabled state on individual tabs not yet wired.

la-upload-box

Single-file drop zone for forms and action flows. Combines click-to-browse and drag-and-drop in one surface with inline states for pending selection, upload progress, and validation errors. Used in the unlock flow document steps and inside la-insight-modal when the resolution is upload-driven.

Props: label (optional field label above the zone), prompt (main zone prompt, defaults to "Click to browse or drag a file here"), hint (secondary line under the prompt), accept (file-type filter — extensions or MIME globs), max-size (bytes), disabled, error, error-message, processing.

Events: la-change (file picked and valid), la-reject (file failed accept or max-size, detail carries reason: 'type' | 'size'), la-remove (user cleared a pending file via the top-right close button).

States: idle · hover · focus-visible · dragover · has-file (pending selection) · processing · error · disabled.

Chrome is deliberately flat. All states share the same background (--la-color-bg-surface) and border colour (--la-color-border-light). State is signalled through the centred lead icon only — neutral upload glyph in idle, green check when a file is picked, red warning in error, grey spinner while uploading. Focus-visible swaps to --la-input-shadow-focus ring (keeps a11y visible despite flat surface).

The upload box is an action surface, not a long-lived document chip.

Single-file only. Multi-file support is not in v1. Stack multiple instances if a form needs several uploads.

Insight improvement aside — design pattern

The aside panel on the legal area page (and other insight-based views) follows a specific pattern: it is an intake coach, not a form.

What it does: Collects context, data, and documents from founders to improve the quality of their insights. Current content: actions to add context, and related documents to browse or upload. Future: step-by-step guided plans.

Mental model: The founder reads their insight (consumption mode), then opens the aside to improve it (intake mode). These modes don't coexist — founders context-switch cleanly.

Design principles:

  • Every item in the aside carries a why — "We're missing your cap table — adding it would improve this insight significantly." Not just a label and a button.
  • Progress and completion states are important — make the founder feel the progress they're making, not the gap that remains.
  • Empty state: inviting and explanatory, never blank.
  • Never feels like a form. The aside earns the interaction by showing the value of each contribution.

Entry point: The trigger to open the aside (on tablet: overlay; on desktop: always visible) should frame the value, not describe the action. "3 ways to strengthen this insight" is better than "Open actions panel."

Tablet behaviour: Overlay drawer (see design-language/layout.md — aside panel at tablet widths). Full-width content when reading, full-attention aside when acting.

Components that live in the aside:

  • la-actions-widget — actions the founder can take to improve the insight
  • la-document-list — related documents that are inputs to the insight

Naming convention

All components are prefixed la- (LawAdvisor). This:

  • Avoids collision with native HTML elements
  • Makes it obvious which elements are custom when reading HTML
  • Groups all library components together in tooling

Component API patterns (Lit)

Properties vs attributes

  • Use @property() for data passed from JS (arrays, objects, booleans)
  • Use @property({ type: String, reflect: true }) when the attribute should be readable in CSS or DevTools
  • Boolean attributes follow HTML spec: presence = true, absence = false

Slots

Use named slots for structural content, default slot for body content:

<la-panel>
  <span slot="title">Panel title</span>
  Content goes here
</la-panel>

Events

Dispatch CustomEvent with bubbles: true for coordination with parent components. Document all events in the component's rationale entry.

Docs pages

Every component has a docs draft at dev/docs-drafts/la-<name>.html that gets promoted to docs-src/components/la-<name>.html by /ready. Iterate the draft live via npm run dev at http://localhost:8000/docs-drafts/la-<name>.html.
