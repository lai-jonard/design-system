Design Language — LawAdvisor AI

This folder documents the design rationale, patterns, and standards for the LawAdvisor AI design system. It is the living reference for how things look and why — used when creating new flows, screens, or components.

When to update

Update this documentation automatically when:

  • A new visual pattern is established (e.g. how we handle empty states)
  • A token decision is made (e.g. a new colour, a border rule)
  • A component behaviour is defined (e.g. how modals animate)
  • A rule is corrected or refined (e.g. "we tried X but switched to Y because...")
  • A spacing or layout standard is set

Do not update for one-off tweaks that are screen-specific and unlikely to generalise.

Relationship to CLAUDE.md and source files

CLAUDE.md at the project root is the canonical foundation — token values, typography rules, animation standards, copy rules, and the Lit component library structure.

This design-language/ folder is the rationale and decision layer on top of it — the why behind decisions, known divergences from defaults, and patterns that emerged from building specific screens and flows. It is a living document updated as the system evolves.

Known divergences from defaults

Topic Default Decision
Disabled state Token-based disabled bg opacity: 0.4 on the element — preserves variant identity
Border distinction Single --border token Two tokens: --la-color-border (interactive) + --la-color-border-light (structural)
Neutral/dark button Not defined --la-color-neutral (#374151) added — dark slate for confirm-weight actions
Heading weight Not specified 600 semibold only — 700 is too heavy for legal product tone
Tab bar border Not specified Removed — padding gap is sufficient separator
Icon weight in notif bars Fill for active/selected Fill used for default type icons in notification bars

When to check which source

  • Building a new component → check CLAUDE.md for tokens and rules, then this folder for architecture patterns
  • Designing a new screen or flow → check layout.md and components.md here
  • Copy / UX writing → CLAUDE.md copy rules section is the source of truth

Core principles

  • Tokens over hardcoding. Every colour, size, radius, and spacing value must use a CSS variable. Never hardcode a hex or pixel value directly in a component.
  • Semibold headings. Weight 600, never 700. Bold is reserved for emphasis within body copy only.
  • Verb-first CTAs. All button labels start with a verb, sentence case. "Approve and Propagate", not "Amendment Approval".
  • Layout vs interactive borders. Two distinct border tokens — lighter for structure, standard for UI controls.
  • Purposeful animation. Motion communicates state change, not decoration. Entrance animations skip on tab navigation.
  • Base before composite. Primitive components live in _base/. Complex, layout-aware components live in _components/.
