Motion

Source of truth

All animation tokens, keyframes, and utility classes live in motion.css at the project root. Never define keyframes or duration values inline — always use the classes and tokens from motion.css.

Link it on every docs page:

<link rel="stylesheet" href="../../motion.css" />

Core rules

  • GPU-safe only. Animate transform and opacity exclusively. Never animate height, width, top, left, background, or color directly.
  • Never use ease-in. Feels sluggish for elements users are watching appear. Use --ease-out for entrances, --ease for hovers and micro-interactions.
  • Never scale from zero. Minimum starting scale is 0.96. Zero-scale entrances look like the UI is popping into existence, not arriving.
  • Max 300ms for repeated interactions. Never exceed 500ms for anything triggered multiple times per session.
  • Stagger: 40–60ms between list items. Above 100ms feels like a slideshow.
  • Always respect prefers-reduced-motion. motion.css collapses the duration vars at :root to 0ms automatically — but only for elements whose styles are in the document tree. Shadow-DOM components must declare their own @media (prefers-reduced-motion: reduce) block inside their static styles, because tokens.ts re-declares the duration vars on :host and shadows the :root override. Either zero the durations on :host under the reduce query, or set transition: none / animation: none on the affected selectors. Pattern reference: la-button, la-modal, la-toast, la-stepper (decision 47 is a stepper-specific call but the reduce block in its source is the canonical example).

Utility classes

Class Use
.anim-fade-up Default entrance — cards, rows, list items on load
.anim-fade-down Alert / notification bars entering from top
.anim-fade-in Simple opacity entrance, no transform
.anim-scale-in Modal / dialog inner panel
.anim-slide-right Panel / drawer sliding in from left
.anim-stagger Parent container — triggers stagger on children
.anim-press Button press feedback

When to use each

Situation Approach
Page load — cards, rows .anim-stagger on parent + .anim-fade-up on each child
Alert / banner from top .anim-fade-down
Modal entrance fade-in on backdrop (JS) + .anim-scale-in on inner panel
Panel / drawer .anim-slide-right
JS-revealed element Remove class → reflow → re-add class
JS-injected list items Add .anim-fade-up + set animationDelay = i * 40 + 'ms'

Re-trigger pattern

el.classList.remove('anim-fade-up');
void el.offsetWidth; // force reflow
el.classList.add('anim-fade-up');

Stagger injected items

items.forEach((el, i) => {
  el.classList.add('anim-fade-up');
  el.style.animationDelay = (i * 40) + 'ms';
});

Entrance sequencing on screens

When building full screens, animate in a deliberate order: chrome → navigation → content.

The header and nav establish where the user is before content arrives. If content appeared first, the page feels unanchored.

Suggested delays:

  • Header: 0ms (immediate)
  • Navigation/tabs: 80ms
  • First content item: 160ms
  • Each subsequent item: +60–80ms stagger

Tab / page navigation

Suppress entrance animations when navigating between tabs or pages — the user expects an instant content swap, not a re-entrance sequence.

// Before navigating
sessionStorage.setItem('la-tab-nav', '1');

// At page load
if (sessionStorage.getItem('la-tab-nav')) {
  sessionStorage.removeItem('la-tab-nav');
  const s = document.createElement('style');
  s.textContent = '*, *::before, *::after { animation-duration: 0ms !important; }';
  document.head.appendChild(s);
  requestAnimationFrame(() => requestAnimationFrame(() => s.remove()));
}
